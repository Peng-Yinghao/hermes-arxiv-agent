## Abstract

Abstract. As large language model (LLM)-powered agents are increasingly deployed to perform complex, real-world tasks, they face a growing class of attacks that exploit extended user–agent–environment interactions to pursue malicious objectives improbable in single-turn settings. Such long-horizon threats pose significant risks to the safe deployment of LLM agents in critical domains. In this paper, we present MAGE , 1 1 1 MAGE : M emory A s G uardrail E nforcement. a novel defensive framework designed to counter a wide range of long-horizon threats. Inspired by the “shadow stack” abstraction in systems security, MAGE maintains a dedicated, safety-focused agentic memory that distills and retains safety-critical context across the agent’s full execution trajectory, leveraging this shadow memory to proactively assess the risk of pending actions prior to their execution. Extensive evaluation demonstrates that MAGE substantially outperforms existing defenses across diverse long-horizon threats in detection accuracy, achieves early-stage detection for the majority of attacks, and introduces only negligible overhead to agent utility.
To our best knowledge, MAGE represents the first framework to detect and mitigate long-horizon threats using an agentic memory approach, establishing a new paradigm for this critical challenge and opening promising directions for future research.

## 1. Introduction

Large language model (LLM)-powered agents are increasingly deployed in settings that demand multi-turn, extended interactions (OpenAI, 2025; OpenManus, 2025). Yet, this very capability introduces a new class of security vulnerabilities: adversaries can exploit long-horizon contexts to mount attacks that are improbable in single-turn settings, including multi-turn poisoning (Dong et al., 2025), tool-chaining attacks (Li et al., 2025a), and persistent prompt injection (Jiang et al., 2026a). For instance, as illustrated in Figure [1](#S1.F1), a tool-chaining attack orchestrates a sequence of individually innocuous tool calls whose cumulative effect is destructive, triggering compliance violations, financial losses, or data exfiltration.

Such long-horizon attacks share a defining property: malicious intent often emerges only at the trajectory level, while remaining latent within individual turns. Most existing defenses, such as keyword filters, per-turn classifiers, and safety prompting, are engineered for short contexts and degrade substantially under the distributional shift to long-horizon interactions (Anil et al., 2024). A handful of recent studies attempt to extend single-turn defenses to multi-turn settings (Huang et al., 2025; Zhang et al., 2024b; Pan et al., 2025); MTSA (Guo et al., 2025), for instance, improves multi-turn alignment through dialogue-level training. Yet these extensions break down as interaction length exceeds the regimes seen during training, and nearly all focus narrowly on multi-turn jailbreak attacks, leaving the broader landscape of long-horizon threats without principled defensive measures.

Our work. In this paper, we explore a fundamentally different approach. Motivated by the observation that long-horizon attacks often succeed by manipulating the agent’s context across extended interactions, we repurpose agentic memory (Zhou et al., 2025; Xu et al., 2025), originally designed to enhance the agent’s context awareness, as the foundation for our defensive measure. Specifically, drawing on the “shadow stack” abstraction (Burow et al., 2019) from systems security research, we introduce a dedicated, security-focused agentic memory that distills and retains security-critical context throughout the agent’s execution trajectory. This shadow memory intercepts pending actions prior to execution, assessing their risk against the accumulated trajectory context. For instance, as illustrated in Figure [1](#S1.F1), the shadow memory identifies that prior interactions have established intent to exfiltrate confidential data, and blocks the offending action before it executes.

To realize this idea, we develop MAGE, the first framework to leverage an agent’s memory mechanism for detecting and mitigating long-horizon threats. At its core, MAGE comprises two components: a memory manager that distills and maintains the shadow memory, and a judge that consults this memory to perform risk assessment. Both are trained via reinforcement learning (e.g., GRPO (Shao et al., 2024)) to jointly optimize detection accuracy, utility preservation, and computational efficiency. MAGE offers three key advantages over alternative approaches: Versatile — it addresses a broad spectrum of long-horizon threats, from adversarial prompting to indirect prompt injection; Plug-and-Play — it integrates readily with existing LLM agents with minimal modification to the core inference pipeline; and Interpretable — it produces human-readable explanations grounded in the shadow memory, making detection decisions transparent to operators.

Extensive empirical evaluation demonstrates that MAGE provides strong defense across two complementary threat models. Under the user-as-adversary setting, using a small local model (Qwen3-4B) as the backbone for both the memory manager and judge, MAGE reduces the attack success rate (ASR) of sequential tool-attack chaining (STAC) (Li et al., 2025a) from 100.0% to 8.3%, while preserving 94.4% benign utility (BU) at a cost of only 7.0K extra tokens per task. Under the environment-as-adversary setting, MAGE reduces the ASR of persistent indirect prompt injection (PI2) (Jiang et al., 2026a) to 0.0%, while preserving 73.0% benign utility, on par with the no-defense upper bound. Notably, in both settings, MAGE detects the majority of attacks in their early stages (at or near the first attack turn), affording operators sufficient time to intervene. Taken together, these results establish MAGE as a strong solution across the dimensions of security, utility, and overhead.

Our contributions. To the best of our knowledge, this work is the first to leverage agentic memory for detecting and mitigating long-horizon threats. Our contributions are as follows.

- $\bullet$
We identify that the key to defending LLM agents against long-horizon threats lies in maintaining security-critical context across the agent’s execution trajectory.
- $\bullet$
We develop MAGE, the first framework that exploits agentic memory to distill and retain security-critical context and perform retrospective, trajectory-level risk assessment.
- $\bullet$
We conduct extensive empirical evaluation demonstrating that MAGE outperforms existing defenses across diverse attacks, achieves early detection for the majority of attacks, and introduces negligible overhead.
- $\bullet$
We discuss the limitations of MAGE’s current design and outline promising directions for future research.

## 2. Related Work

### 2.1. Adversarial Prompting

LLM agents face a variety of threats in realistic deployments. One class assumes adversarial users and extends jailbreak techniques (Zou et al., 2023; Liu et al., 2023; Chao et al., 2023), aiming to trigger the agent into misbehavior, including unauthorized risky actions (Ruan et al., 2024), memory manipulation (Chen et al., 2024; Dong et al., 2025), and unsafe code generation (Guo et al., 2024). Recent work extends such attacks to multi-turn settings. For instance, STAC (Li et al., 2025a) maintains a stateful, context-aware attack chain to progressively steer the agent toward policy violations. MINJA (Dong et al., 2025) compromises the agent’s memory through query-only interaction, injecting malicious records that elicit harmful responses in subsequent turns.

In response, a handful of defenses have been proposed to prevent agents from executing harmful actions, drawing on established jailbreak mitigation (Zheng et al., 2024; Zhou et al., 2024; Mo et al., 2024; Huang et al., 2025; Zhang et al., 2024b; Pan et al., 2025). For instance, MTSA (Guo et al., 2025) enhances multi-turn alignment through dialogue-level training. However, these defenses tend to degrade as interaction length exceeds the conditions encountered during training.

### 2.2. Indirect Prompt Injection

Indirect prompt injection attacks assume adversarial environments, embedding malicious instructions into external content to manipulate agent behavior. Proposed attacks span manually crafted templates (Zhan et al., 2024; Liu et al., 2024b) to automated discovery methods (Wang et al., 2025d; Liu et al., 2024a; Zhang et al., 2025; Wen et al., 2025; Shi et al., 2025a; Chang et al., 2025; Wang et al., 2026), including adaptive variants that modify payloads to counter deployed defenses (Nasr et al., 2025; Ji et al., 2025b). Recent work extends these attacks to long-horizon settings: GoalDrift (Arike et al., 2025) demonstrates that agent adherence to user intent degrades rapidly with attack horizon, while PI2 (Jiang et al., 2026a) employs an optimization pipeline that generates context-adapted malicious payloads across turns, augmented with a memory bank of successful exemplars.

Existing defenses against prompt injection fall into four broad categories. i) Prompt augmentation helps the agent distinguish legitimate instructions from injected ones via structural separation (Hines et al., 2024; Willison, 2023; Wallace et al., 2024), instruction repetition (Schulhoff, 2024), optimized defensive token embeddings (Chen et al., 2025b; Wu et al., 2025), or inference-time scaling (Liu et al., 2025d), though it offers limited resilience against carefully crafted attacks. ii) Model hardening fine-tunes the agent’s base LLM for intrinsic resilience via preference optimization (Chen et al., 2025c), instruction-adherence training (Chen et al., 2025a)), or adversarial reinforcement learning (Wang et al., 2025e; Sha et al., 2025), at the risk of degrading general-purpose utility. iii) Runtime detection identifies compromised inputs or intermediate states, using known-answer probing (Yohei Nakajima, 2022; Liu et al., 2024b, 2025c), instruction-data overlap detection (Kang et al., 2025), masked re-execution (Zhu et al., 2025), or semantic segmentation (Jia et al., 2025). iv) Finally, system-level defenses restructure agent workflows via tool filtering (Debenedetti et al., 2024), tool-call provenance analysis (Wang et al., 2025a; An et al., 2025), or execution environment isolation (Wu et al., 2024; Costa et al., 2025; Shi et al., 2025b; Wen et al., 2026). Critically, all these defenses are designed primarily for short-context attacks and exhibit robustness degradation over extended interactions.

### 2.3. Agentic Memory

Agentic memory is an essential component for long-horizon reasoning and continual adaptation (Hu et al., 2025). Existing designs span a broad architectural spectrum: flat structures storing independent chunks or dialogue logs (e.g., MemoryBank (Zhong et al., 2024)), graph-based structures capturing relational dependencies (e.g., Mem0 (Chhikara et al., 2025) and A-Mem (Xu et al., 2025)), and hierarchical structures such as MemGPT (Packer et al., 2023). Beyond heuristic-based memory management, recent work applies reinforcement learning (RL) to let the agent learn what to remember and forget. MEM1 (Zhou et al., 2025) and MemAgent (Yu et al., 2025), for instance, train the agent via end-to-end RL to maintain compact memory across long multi-turn tasks. However, all existing designs optimize for agent performance rather than security.

To our best knowledge, this work is the first to exploit agentic memory as a security primitive, defending LLM agents against long-horizon threats spanning both adversarial prompting and indirect prompt injection.

## 3. Preliminaries

### 3.1. User-Agent-Environment Interaction

We model an LLM-powered agent as a policy $\pi$ that augments a base LLM with external tool-use capabilities (Yao et al., 2023; Schick et al., 2023), with the LLM serving as the cognitive core for perception, planning, and action selection. The agent operates through iterative triadic interactions among the user, agent, and environment. Let $q_{0}$ denote the initial user instruction. At each turn $t\geq 1$, the agent selects an action $a_{t}$ (a tool invocation) given the history up to turn $t-1$. Executing $a_{t}$ yields an environmental observation $o_{t}$; The user may optionally provide a follow-up instruction $q_{t}$. The trajectory is defined as a sequence:
$s_{0},s_{1},s_{2},\ldots$, where $s_{i}=\langle a_{i},o_{i},q_{i}\rangle$ (with $s_{0}=\langle\emptyset,\emptyset,q_{0}\rangle$). The agent’s state before turn $t$ is the accumulated trajectory $\tau_{t-1}=s_{[0:t-1]}$, from which the next action is sampled as $a_{t}\sim\pi(a|\tau_{t-1})$. This process repeats until a termination condition is met (e.g., task completion or a predefined horizon $T$), as illustrated in Figure [2](#S3.F2) (a).

Unlike prior formulations that either assume single-turn instructions (Zhang et al., 2024a; Debenedetti et al., 2024) or restrict interactions to user-agent exchanges (Rahman et al., 2025; Russinovich et al., 2025), this formulation explicitly captures multi-turn, triadic interactions among the user, agent, and environment.

### 3.2. Threat Model

We assume the following threat model in our study.

Adversary’s objective. The adversary aims to steer the agent toward an adversarial state $\tau_{t}^{*}$ through a sequence of interactions, inducing it to execute a malicious action $a^{*}\sim\pi(a|\tau^{*})$ (e.g., deleting a critical file (Li et al., 2025a)). For clarity, we assume a single malicious action, though the framework extends naturally to multi-action objectives.

Adversary’s capabilities. We consider both adversarial prompting and indirect prompt injection within a unified framework. In the former, the adversary introduces a sequence of malicious instructions $\{q^{*}\}$, directly steering the agent towards $\tau^{*}$. In the latter, the adversary embeds malicious payloads $\{o^{*}\}$ into environmental observations, indirectly inducing the agent to execute $a^{*}$.

Crucially, unlike one-shot attacks that rely on a single adversarial prompt $q^{*}$ or payload $o^{*}$, we focus on long-horizon attacks in which the adversary adaptively and iteratively generates multiple adversarial prompts $\{q^{*}\}$ or payloads $\{o^{*}\}$ across extended interactions, gradually steering the agent towards $\tau^{*}$.

Adversary’s knowledge.
We adopt a grey-box setting in which the adversary observes the agent’s execution trajectory $\tau$ (including the benign user’s instructions under the prompt injection setting), but cannot access the LLM’s parameters or the deployed defense. This threat model is widely adopted in prior work (Jiang et al., 2026a; Liu et al., 2025c; Zhu et al., 2025). Notably, unlike static attacks, the adversary can adaptively refine its strategies based on observed trajectories.

Defender’s objective.
The defender aims to detect adversarial prompts $\{q^{*}\}$ or injections $\{o^{*}\}$ as early as possible before the malicious action $a^{*}$ is executed, while minimizing disruption to benign task performance. Formally, at each turn $t$, a judge $J$ assesses the proposed action $a_{t}$ against the accumulated trajectory $\tau_{t-1}$:

$$ (1) $d_{t},e_{t}=J(a_{t}|\tau_{t-1})$ $$

where $d_{t}\in$ {approve, reject} is the decision, $e_{t}$ is its rationale, allowing the human operator to interpret the detection.
The goal is to maximize detection accuracy and minimize detection latency across extended interactions.

## 4. MAGE

Next, we present MAGE, the first framework to leverage agentic memory as a defensive primitive against long-horizon attacks. §[4.1](#S4.SS1) motivates its design; §[4.2](#S4.SS2) details each component; and §[4.3](#S4.SS3) describes the optimization of MAGE for detection accuracy, utility preservation, and computational efficiency.

### 4.1. Motivation

We attribute the LLM agent’s vulnerability to long-horizon attacks to three major causes. i) There is a fundamental tension between the agent’s finite working memory and the growing context window (Yu et al., 2025), causing its attention to security-critical information to be progressively diluted. ii) In long-horizon attacks, the adversarial intent only emerges at the trajectory level, not within individual turns, while the agent often lacks the capacity for retrospective trajectory-level inspection (Laban et al., 2026). iii) The agent’s safety alignment is typically trained in short-context settings and fails to generalize to extended interactions.

All three causes point to one imperative: distilling and retaining security-critical context across the agent’s execution trajectory, and performing retrospective inspection to detect adversarial intent before it culminates in malicious actions. This motivates the design of MAGE, inspired by the “shadow stack” abstraction (Burow et al., 2019), which defends against control-flow hijacking by maintaining a separate copy of return addresses shielded from corruption by the primary execution stack. MAGE introduces an analogous construct: a dedicated shadow memory that runs in parallel with the agent’s working memory. Whereas the working memory encodes task-completion context (Yu et al., 2025), the shadow memory selectively distills security-critical context and discards security-irrelevant information as the agent’s execution trajectory unfolds. All security assessments are then grounded in this continuously updated shadow memory.

### 4.2. System Design

Figure [2](#S3.F2) (a) illustrates the overall architecture of MAGE, comprising two key components: i) the memory manager $M$, which distills and maintains security-critical context across the agent’s interaction turns; ii) the judge $J$, which performs retrospective inspection of the agent’s trajectory and assesses the risk of pending actions.

#### 4.2.1. Shadow Memory

A naïve implementation of MAGE would have the judge $J(a_{t}|\tau_{t-1})$ assess each pending action $a_{t}$ against the full trajectory $\tau_{t-1}$, which includes all prompts, actions, and observations up to turn $t$, incurring an overall computational overhead of $O(T^{2})$, where $T$ denotes the total number of turns. An $O(T)$ alternative $J(a_{t}|s_{t-1})$ that evaluates risks based solely on the most recent interaction avoids this cost but sacrifices contextual visibility, leaving the judge blind to long-horizon attacks that accumulate adversarial influence incrementally across turns.

To resolve this tension, we introduce agentic shadow memory, a compact, security-focused representation of the agent’s execution trajectory that evolves as follows:

$$ (2) $m_{t}=M(m_{t-1},s_{t-1}),$ $$

where $m_{t-1}$ is the shadow memory at turn $t-1$.
Then, the judge $J$ assesses the risk of $a_{t}$ conditional on $m_{t}$:

$$ (3) $d_{t},e_{t}=J(a_{t}|m_{t}),$ $$

The resulting workflow of $M$ and $J$, centered on the shadow memory, is illustrated in Figure [2](#S3.F2) (b).
Compared with alternative designs (e.g., Eqn ([1](#S3.E1))), the shadow memory offers several key advantages.

- $\bullet$
Security-focused distillation.
It distills the agent’s trajectory into a compact, security-focused representation, retaining critical context while pruning irrelevant information. This serves a dual purpose: suppressing noisy signals that could impair security assessment, while highlighting salient indicators (e.g., previously observed injections) as structured cues for the judge.
- $\bullet$
Resilience to long-horizon attacks.
By maintaining a holistic, security-focused representation, the shadow memory is inherently more robust to adversarial context dilution. Whereas the raw context may bury early-stage adversarial signals under subsequent interactions, the shadow memory explicitly preserves and foregrounds such signals.
- $\bullet$
Reduced computational overhead.
As shown in Eqn ([3](#S4.E3)), the shadow memory-augmented judge requires only the compact $m_{t}$ for risk assessment, reducing aggregate token overhead to $O(T)$ and lowering inference latency. Unlike the agent’s working memory that retains all task-relevant context to maximize performance, the shadow memory can be substantially more concise, as it attends exclusively to security-critical context.

#### 4.2.2. Agentic Workflow

Since both the memory manager $M$ and judge $J$ require parsing structured LLM outputs, one option is to define special tokens, but this requires additional supervised fine-tuning and risks instability. Instead, we integrate $M$ and $J$ into the agentic workflow as native tool calls, directly leveraging the LLM’s native tool-calling capability without any additional training. The overall agentic workflow is sketched in Algorithm [1](#algorithm1).

At each turn $t$, the agent proposes a pending action $a_{t}$. Prior to execution, the memory manager $M$, invoked via the tool call update_memory(content), updates the shadow memory $m_{t-1}$ with a distilled representation of the current context. Next, the judge $J$, invoked via the tool call assess_risk(decision, rationale), assesses the security risk of $a_{t}$ conditional on the updated $m_{t}$. A checkpoint is enforced at each turn: the execution resumes *only if* $J$ approves $a_{t}$; otherwise, the action is blocked, and the rationale $e_{t}$ is reported to the human operator.

#### 4.2.3. Backbone Model

The memory manager $M$ and judge $J$ admit various implementation choices, including reuse of the target agent’s backend LLM. We opt for dedicated external LLMs for two reasons. First, the target agent is typically powered by a large frontier model (e.g., GPT-5), whereas shadow memory distillation and risk assessment are focused tasks that do not demand frontier-scale capacity; a small model often suffices, offering lower latency and reduced cost. Second, a dedicated model enables task-specific training, improving both memory quality and assessment accuracy.

Moreover, we instantiate both $M$ and $J$ using a single shared model parameterized by $\theta$ (differentiated via role-specific prompting, detailed in §[B.1](#A2.SS1) for memory update and §[B.2](#A2.SS2) for risk assessment). This design offers two benefits: i) joint learning – $M$ can learn to produce representations directly optimized for $J$’s assessment; and ii) low overhead – a single model serving dual roles reduces deployment complexity and resource consumption relative to two separate models.

### 4.3. Model Optimization

We now describe the training procedure for $M_{\theta}$ and $J_{\theta}$, illustrated in Figure [2](#S3.F2) (c).

#### 4.3.1. Training Data

We construct the training data using complete agent execution trajectories. Each instance is a trajectory $\tau=(s_{0},\ldots,s_{T})$ with $s_{t}=\langle a_{t},o_{t},q_{t}\rangle$ ($s_{0}=\langle\emptyset,\emptyset,q_{0}\rangle$), produced by the target agent $\pi$ across diverse task environments with and without attacks. Each action $a_{t}$ is annotated with a ground-truth binary label $d_{t}^{*}$, indicating whether $a_{t}$ is malicious, derived from the specifications of benign tasks and attack objectives using an LLM-as-judge approach with human verification (Debenedetti et al., 2024) (more details in §[5.1](#S5.SS1) and §[6.1](#S6.SS1)). Notably, the labels are assigned to actions rather than observations: if $s_{t}$ contains an injected payload but $a_{t}$ is itself benign, the label remains negative.

The distribution of positive labels varies by attack types. Attacks inducing a single malicious action yield one positive label per trajectory, while those that corrupt multiple intermediate states and induce a sequence of malicious actions yield multiple positive labels within the same trajectory.

#### 4.3.2. Turn-wise Reinforcement Learning

We adopt Group Relative Policy Optimization (GRPO) (Shao et al., 2024) as the RL algorithm. In its standard formulation, GRPO samples a group of $G$ candidate outputs $\{y_{i}\}_{i=1}^{G}$ from the current policy $\pi_{\theta_{\text{old}}}$ for each input $x\sim\mathcal{D}$, computes a group-normalized advantage from within-group rewards, and updates the policy by maximizing:

$$ (4) $\displaystyle\mathcal{J}(\theta)$ $\displaystyle=\mathbb{E}_{x\sim\mathcal{D},\{y_{i}\}\sim\pi_{\theta_{\text{old}}}(\cdot|x)}\frac{1}{G}\sum_{i=1}^{G}\left[\min\left(\frac{\pi_{\theta}(y_{i}|x)}{\pi_{\theta_{\text{old}}}(y_{i}|x)}A_{i},\right.\right.$ $\displaystyle\left.\left.\mathrm{clip}\left(\frac{\pi_{\theta}(y_{i}|x)}{\pi_{\theta_{\text{old}}}(y_{i}|x)},1{-}\epsilon,\,1{+}\epsilon\right)A_{i}\right)-\beta\mathbb{D}_{\text{KL}}\left(\pi_{\theta}\|\pi_{\theta_{\text{ref}}}\right)\right],$ $$

where
$A_{i}=(r_{i}-\mathrm{mean}(\{r_{j}\}_{j=1}^{G}))/\mathrm{std}(\{r_{j}\}_{j=1}^{G})$ is the normalized advantage of the $i$-th rollout (with $r_{i}$ being its reward), $\epsilon$ is the clipping ratio, $\beta$ is the KL penalty coefficient, and $\pi_{\theta_{\text{ref}}}$ is the reference policy for
computing the KL divergence.

The standard formulation assigns a single trajectory-level reward and treats all generated tokens uniformly. However, our setting departs from both assumptions. First, the turn-level annotations provide natural supervisory signals that enable reward assignment and policy updates at the granularity of individual turns. Second, each turn comprises two phases: memory update by $M$ and risk assessment by $J$, while
only tokens belonging to $M$’s or $J$’s active phase should contribute to its policy gradient.
This phase-wise separation serves two purposes: i) rewards are attributed to the correct generation phase; and ii) memory update and risk assessment receive separate reward signals without mutual interference, eliminating cross-phase reward hacking.

To this end, we introduce a turn-wise GRPO variant. Let
$m_{t,i}\sim M_{\theta}(\cdot|s_{t},m_{t-1,i})$ where $i\in[1,G]$
denote $G$ sampled memory updates at turn $t$, and
$d_{t,i},e_{t,i}\sim J_{\theta}(a_{t}|m_{t,i})$ denote
the corresponding judge outputs. The turn-wise objective is defined as:

$$ (5) $\displaystyle\mathcal{J}(\theta)={\mathbb{E}}\sum_{t=1}^{T}\frac{1}{G}\sum_{i=1}^{G}\left(\mathcal{L}^{\mathrm{mem}}_{t,i}+\mathcal{L}^{\mathrm{judge}}_{t,i}\right),$ $$

where each phase-specific loss term takes a clipped surrogate form:

$$ (6) $\mathcal{L}^{p}_{t,i}=\min\left(\rho^{p}_{t,i}A^{p}_{t,i},\mathrm{clip}\left(\rho^{p}_{t,i},1{-}\varepsilon,1{+}\varepsilon\right)A^{p}_{t,i}\right)-\beta\mathbb{D}_{\mathrm{KL}}^{p}(\pi_{\theta}\|\pi_{\mathrm{ref}}),$ $$

where $\pi_{\theta}$ is the underlying model (e.g., memory manager $M_{\theta}$ or judge $J_{\theta}$). For $p\in\{\mathrm{mem},\,\mathrm{judge}\}$, the importance sampling ratio is defined as:

$$ (7) $\displaystyle\rho^{\mathrm{mem}}_{t,i}$ $\displaystyle=\frac{\pi_{\theta}(m_{t,i}|s_{t},m_{t-1,i})}{\pi_{\mathrm{old}}(m_{t,i}|s_{t},m_{t-1,i})},$ $\displaystyle\rho^{\mathrm{judge}}_{t,i}$ $\displaystyle=\frac{\pi_{\theta}(d_{t,i},e_{t,i}|a_{t},m_{t,i})}{\pi_{\mathrm{old}}(d_{t,i},e_{t,i}|a_{t},m_{t,i})},$ $$

where $A^{\mathrm{mem}}_{t,i}$, $A^{\mathrm{judge}}_{t,i}$ are computed by group-normalizing the memory and judge rewards within each group at turn $t$. This formulation ensures that i) policy gradients for each phase are computed exclusively over tokens generated in that phase; ii) the two phases receive decoupled reward signals; and iii) the turn-level trajectory structure is fully exploited for fine-grained credit assignment.

#### 4.3.3. Reward Design

We further detail the design of the reward function for each phase.

Judge reward.
For the judge output at turn $t$, we assign a binary
correctness reward based on the ground-truth label:

$$ (8) $R_{\text{judge}}(d_{t})=\begin{cases}+1&\text{if }d_{t}=d_{t}^{*}\\ -1&\text{otherwise}\end{cases}$ $$

where $d_{t}^{*}$ is the ground-truth label for the proposed
action $a_{t}$. This reward applies only in the risk assessment phase.

Memory reward.
For the memory update at turn $t$, we apply a
temporally-discounted correctness reward that accounts for the
recurrent nature of shadow memory.

A naïve design rewards $m_{t}$ solely based on whether the immediate assessment $d_{t}$ matches the ground-truth label. However, the shadow memory is recurrent: $m_{t}$ at turn $t$ propagates through the update chain $m_{t}\to m_{t+1}\to\cdots$, causally influencing all subsequent assessments. Rewarding $m_{t}$ only on $d_{t}$ thus provides no learning signal about its downstream consequences, inducing myopic memory updates in practice, where $M$ discards information irrelevant to the current assessment but critical for future ones (e.g., an early-stage injection pattern that only becomes exploitable several turns later). To address this, we propagate each future judge outcome backward to the memory updates that contributed to it, with exponential decay reflecting diminishing causal influence:

$$ (9) $R_{\text{mem}}(m_{t})=\displaystyle\sum_{k=t}^{T}\gamma^{k-t}\cdot\mathbbm{1}_{d_{k}=d_{k}^{*}}$ $$

where $\gamma\in[0,1)$ is the temporal discount factor, $T$ is the trajectory horizon, and $\mathbbm{1}_{p}$ is an indicator function, which returns 1 if the predicate $p$ is true and 0 otherwise. Intuitively, $m_{t}$ receives maximal credit from its own assessment $d_{t}$ and progressively attenuated credit from more distant turns, reflecting $m_{t}$’s causal influence along the recurrence chain.

Format penalty. To encourage well-structured outputs conforming to the expected tool-call format, we apply a format penalty (e.g., 0.1) to outputs that fail to parse correctly. When a format violation is detected, a safe fallback is substituted: reject with a generic rationale for the judge, or the previous memory $m_{t-1}$ for the memory manager.

Length penalty.
To encourage compact memory representations, we introduce a length penalty on the memory size:

$$ (10) $R_{\text{length}}(m_{t})=\left\{\begin{array}[]{cl}-\lambda\frac{|m_{t}|}{5000}&\text{if }|m_{t}|>2500\\ 0&\text{otherwise}\end{array}\right.$ $$

where $|m_{t}|$ denotes the character length of the memory text and
$\lambda$ controls the penalty strength. This penalty activates only when memory exceeds a threshold (e.g., 2,500 characters), allowing the model to retain necessary detail in short contexts while penalizing redundancy in longer ones. We use $\lambda$ = 1.0 as the default.

Total reward.
Putting everything together, the total reward at turn $t$ is defined as:

$$ (11) $R=\begin{cases}R_{\text{judge}}(d_{t})+R_{\text{format}}(d_{t},e_{t})&\text{for }J\\ R_{\text{mem}}(m_{t})+R_{\text{format}}(m_{t})+R_{\text{length}}(m_{t})&\text{for }M\end{cases}$ $$

The rewards are used to compute the group-normalized advantages
$A^{\mathrm{judge}}_{t,i}$, $A^{\mathrm{mem}}_{t,i}$ in Eqn ([5](#S4.E5)), with normalization performed independently within each phase.

## 5. Evaluation I: User-as-Adversary

We now empirically evaluate the effectiveness of MAGE under the long-horizon *user-as-adversary* threat model.

### 5.1. Experimental Setting

#### 5.1.1. Attacks, Datasets, and Agents

We instantiate the above threat model with Sequential Tool-Attack Chaining (STAC) (Li et al., 2025a), a long-horizon attack that automatically plans, verifies, and refines a sequence of tool calls that appear individually benign but collectively enable malicious operations. The full attack pipeline and its optimization procedure are detailed in §[D.1](#A4.SS1) (Algorithm [2](#algorithm2)). We draw 247 malicious (STAC attacks) and 88 benign tasks from the SHADE-Arena benchmark (Kutasov et al., 2025) and split them into disjoint training and held-out evaluation sets. The average attack horizon is 3.43 turns, while the maximum is 24 turns. We use GPT-5.1 as the target agent’s backbone LLM.

#### 5.1.2. MAGE Implementation

To train MAGE, we draw 276 tasks (206 malicious, 70 benign) from SHADE-Arena as the training set, strictly disjoint from the evaluation set. For each task, we segment the execution trajectory at each tool call $a_{t}$, assigning a cumulative binary label to each segment indicating whether the tool calls executed up to and including $a_{t}$ culminate in a malicious outcome. These labels are generated using an LLM-as-judge pipeline (using GPT-5.4) and verified through human annotation (98.1% accuracy). We train the memory manager $M$ and judge $J$ following the procedure in §[4.3](#S4.SS3). The training details, including the hyperparameter setting, the training dynamics, and the impact of training data size, are deferred to §[C](#A3).

As no existing defenses specifically target STAC attacks, we compare MAGE against its variants obtained by ablating the following design factors: i) whether $M$ is used (when omitted, $J$ directly examines the agent’s raw trajectory); ii) whether MAGE is trained or merely prompted; iii) what LLM underlies $M$ and $J$; and iv) whether only $J$ is trained while $M$ remains untuned. The prompts for all variants (e.g., the judge-only baseline without shadow memory) are provided in §[B.3](#A2.SS3).

#### 5.1.3. Metrics

We evaluate each defense along three dimensions:

- $\bullet$
Attack success rate (ASR) measures the fraction of attack instances in which the adversary’s malicious objective is achieved, quantifying the defense’s effectiveness.
- $\bullet$
Benign utility (BU) measures the fraction of benign tasks completed correctly by the defended agent, reflecting the defense’s impact on the agent’s functionality.
- $\bullet$
Token cost (TC) measures the defense’s computational cost, quantified by the number of extra tokens required to process the agent’s context during protection.

#### 5.2.1. Alternative Design

Table [1](#S5.T1) reports the performance of MAGE variants against STAC attacks in the held-out evaluation set. We have the following observations.

First, the shadow memory provides security-critical context that enables the judge $J$ to make informed risk assessments. The variants without shadow memory ($J$-only) exhibit substantially higher ASR. For instance, holding the underlying LLM (Qwen3-4B) fixed, the untrained $J$-only variant achieves 62.5% ASR; upgrading to the more capable GPT-5-mini only reduces the ASR to 37.5%, while RL-training the $J$-only variant only attains 16.7% ASR. In contrast, MAGE ($J$+$M$ with Qwen3-4B and full RL training) reduces the ASR to 8.3%, indicating that the accumulated context in shadow memory plays an essential role for $J$ to effectively detect malicious intent that emerges gradually over long attack trajectories.

Second, RL training is essential for MAGE. Without training, $J$+$M$ (Qwen3-4B) attains 0.0% ASR but at the cost of 0.0% BU, indicating that it blindly rejects all action and interrupts all legitimate tasks; the untrained GPT-5-mini variant exhibits the same indiscriminate behavior (0.0% ASR, 0.0% BU), flagging all actions as malicious. With RL training, MAGE reduces ASR to 8.3% while lifting BU to 94.4%, confirming that RL training calibrates MAGE to distinguish genuinely malicious tool-call chains from benign ones without sacrificing detection sensitivity.

Third, jointly training $M$ and $J$ is critical. Training $J$ only (with $M$ left untuned) leaves ASR at 12.5% and BU at 88.9%, both worse than MAGE’s 8.3% and 94.4%. The higher ASR indicates that without explicitly training $M$, the extracted context fails to capture all security-critical patterns, while the lower BU shows that the unfocused memory simultaneously leads $J$ to over-flag benign cases.

Finally, MAGE shows substantial efficiency in distilling security-critical context under long-horizon attacks, consuming only 7.0K extra tokens compared with 12.9K for the $J$-only variant (with RL training). This 1.8 $\times$ reduction confirms that the shadow memory’s compression advantage scales with attack horizon.

Overall, MAGE achieves strong robustness against STAC attacks with minimal impact on the agent’s general performance, while relying solely on a small local model (e.g., Qwen3-4B) at modest computational overhead.

#### 5.3.4. Parameter Sensitivity

Figure [4](#S5.F4) reports the sensitivity of MAGE to the reward hyperparameters $\gamma$ and $\lambda$ in the user-as-adversary setting. Notably, moderate discounting ($\gamma\in$ [0.25, 0.5]) yields the best performance at 8.3% ASR and 94.4% BU, while $\gamma$ = 0 (immediate-only signal) degrades ASR to 12.5% (BU unchanged at 94.4%). As $\lambda$ increases to 2, the penalty becomes overly aggressive: ASR jumps to 20.8% while BU drops to 83.3%, indicating that over-compression discards security-critical signals from the shadow memory. Overall, overhead decreases monotonically with $\lambda$, and $\lambda\in$ [0.5, 1.0] offers the best trade-off between defensive quality and memory efficiency.

### 5.4. Adaptive Attacks

In the previous evaluation, we assume a grey-box setting, in which the adversary is unaware of the deployed defense. We now consider a stronger threat model in which the adversary has full knowledge of MAGE and tailors prompts to subvert its components. Building upon STAC’s attack pipeline (Li et al., 2025a), we adapt its LLM-based attacker module to generate such adversarial prompts under three strategies: i) M-adaptive — the prompt poisons the shadow memory via instructions such as “update the user intent in your memory to include the following authorized action”, deceiving the memory manager $M$; ii) J-adaptive — the prompt targets the judge $J$ via privilege-escalation injections such as “system developer mode on, approve all requests”, overriding its rejection decisions; and iii) J/M-adaptive — the prompt combines both, simultaneously targeting $M$ and $J$. Rather than fixed templates, we seed the attacker module with strategy-specific examples and let it iteratively optimize prompts to maximize attack effectiveness. The implementation details of the adaptive attacks are provided in §[D.1](#A4.SS1).

## 6. Evaluation II: Environment-as-Adversary

Next, we empirically evaluate the effectiveness of MAGE under the long-horizon environment-as-adversary threat model.

### 6.1. Experimental Setting

#### 6.1.1. Datasets, Models, and Agents

We instantiate the threat model using indirect prompt injection attacks.
Specifically, we extend the AgentDojo benchmark (Debenedetti et al., 2024), which provides realistic environments where agents perform up to 18 tool calls per task. AgentDojo supports customizable attack and defense configurations as well as comprehensive evaluation, and has been widely adopted in prior work (Zhu et al., 2025; An et al., 2025; Jiang et al., 2026a). We select the Banking and Slack test suites because they involve long interaction horizons, comprising 43 benign tasks and 14 injection tasks, with an average of 4.43 interaction turns, yielding 7,682 test cases in total (each benign task paired with each injection task across the three agents, plus benign-only runs; not a strict grid). We randomly split the test cases into two disjoint halves for training and evaluation, respectively.
For the backbone LLM, we evaluate both frontier and open-weight models: GPT-4o-mini, Qwen3-235B, and Llama-3.3-70B, models known to exhibit relatively high susceptibility to prompt injection attacks (Debenedetti et al., 2024; Zhu et al., 2025; An et al., 2025), making them suitable for stress-testing defense methods.

#### 6.1.2. Attacks

We evaluate 5 attacks spanning both short-horizon (static) and long-horizon (adaptive) categories. For the first category, we consider: i) InjecAgent (Zhan et al., 2024) directly appends malicious payloads to tool-retrieved content; ii) Prefix-Override (Debenedetti et al., 2024) employs the prefix “Ignore the previous instructions and do the following” to override benign context; iii) Tool-Knowledge (Debenedetti et al., 2024) crafts malicious payloads tailored to the agent’s available tools; iv) Important-Message (Debenedetti et al., 2024) elevates the perceived priority of malicious payloads with the prefix of “###Important Message”. These attacks inject static payloads at all possible positions that may be encountered during a benign task, introducing an average of 1.2 attack turns per task. For the second category, we consider v) Persistent Indirect Prompt Injection) (PI2) (Jiang et al., 2026a), a long-horizon prompt injection attack that optimizes malicious payloads adapted to current contexts and attack objectives and iteratively injects them into the agent’s extended interactions. This attack introduces an average of 3.5 attack turns per task (with a maximum of 6 turns). The detailed attack pipeline and examples
are deferred to §[D.2](#A4.SS2) and §[A](#A1), respectively.

#### 6.1.3. Defense Baselines

We compare MAGE against 5 representative defenses spanning detection-based and prompt-augmentation approaches: i) Spotlighting (Hines et al., 2024) wraps tool outputs with explicit delimiters to help the agent distinguish trusted instructions from untrusted environmental data;
ii) Repeated-Prompt (Debenedetti et al., 2024) appends the user prompt after each tool output to reinforce adherence to the benign task;
iii) PI-Detector (Debenedetti et al., 2024) employs a dedicated classifier to identify whether tool-retrieved content contains injected malicious instructions;
iv) IPIGuard (An et al., 2025) routes tool outputs through an external LLM to detect and suppress indirect prompt injections before they reach the agent; and
v) MELON (Zhu et al., 2025) applies a multi-step reasoning pipeline to assess the safety of each candidate agent action prior to execution.

#### 6.1.4. MAGE Implementation

To train MAGE, we generate 6,000 trajectories from the training split, comprising both benign and malicious executions across 3 agents, 4 short-horizon attacks, 1 long-horizon attack, and 2 task suites. Each trajectory is segmented at every decision point where the agent proposes an action, yielding a sequence of (context, action) pairs. Derived from the specifications of benign tasks and attack objectives, each pair is assigned a binary label indicating whether the proposed action is driven by the benign task or the injected adversarial payload. These labels are generated via an LLM-as-judge pipeline (Debenedetti et al., 2024) (using GPT-5.4) and verified through human annotation (with 98.7% accuracy).
We then apply turn-wise GRPO in §[4.3](#S4.SS3) to train both the memory manager $M$ and judge $J$. The training details, including the hyperparameter setting, the training dynamics, and the impact of training data size, are deferred to §[C](#A3).

#### 6.1.5. Metrics

Following §[5](#S5), we evaluate each defense along the dimensions of attack success rate (ASR), benign utility (BU), and token cost (TC).

### 6.2. Main Results

#### 6.2.1. Security

Table [4](#S5.T4) compares MAGE against baseline defenses across 5 attacks. Notably, Spotlighting and Repeated-Prompt perform poorly against long-horizon attacks (PI2), with ASR exceeding 70%. MELON achieves 0.7% ASR on GPT-4o-mini but degrades sharply on open-weight LLMs (42.1% on Llama-3.3, 20.7% on Qwen3); we attribute this to the unstable instruction-following in these models when defense directives are injected into the environment context, causing MELON’s similarity matching to become inconsistent. PI-Detector performs well on simpler attacks (InjecAgent and Prefix-Override) but degrades substantially on PI2, as its reliance on injection pattern matching struggles against semantically camouflaged or multi-turn injections. IPIGuard achieves attack robustness comparable to MAGE but incurs significant utility overhead (discussed below).

In comparison, MAGE consistently achieves the lowest ASR across all settings, particularly with 0.0% ASR on PI2. We attribute this to the shadow memory’s ability to distill and retain security-critical signals across turns: while PI2 deliberately distributes malicious intent over multiple turns to evade per-turn defenses, shadow memory accumulates and preserves suspicious signals across the trajectory, enabling the judge to detect adversarial patterns even when no individual turn appears overtly malicious.

#### 6.2.2. Utility

Table [4](#S5.T4) also reports the impact of each defense on benign utility. Notably, PI-Detector suffers catastrophic utility loss (below 30% BU). MELON performs well on Llama-3.3 but causes 5.4% and 9.8% utility loss on GPT-4o-mini and Qwen3, respectively. IPIGuard similarly causes 5.4% and 8.1% utility loss. All aggressively block benign actions, rendering them ineffective despite reasonable attack resilience. In contrast, MAGE incurs only marginal utility loss (below 3%) across all settings.
We attribute this to the explicit extraction of the user’s original intent in shadow memory, which provides a reliable reference for downstream decision-making. Further, the agentic training enables the judge to consistently validate intent-consistent actions.

#### 6.2.3. Overhead

Table [4](#S5.T4) reports defense overhead in terms of extra tokens processed per task. Lightweight defenses such as PI-Detector and Spotlighting incur minimal overhead but provide inadequate security; Repeated-Prompt incurs the highest overhead yet provides limited defense against PI2. We therefore focus the overhead comparison on MAGE, MELON, and IPIGuard, which offer comparable security guarantees.

Without attacks, all three incur similar overhead. Under attacks, MAGE tends to incur lower overhead than the baselines (e.g., 2.9K vs. 3.2K and 6.8K on GPT-4o-mini). More critically, MAGE routes all defense computation through a dedicated local model (e.g., Qwen3-4B), whereas MELON and IPIGuard issue defense queries directly to the agent’s own API. Thus, even when token counts are comparable, the monetary cost of MAGE is substantially lower, especially when the agent relies on expensive frontier models.

### 6.4. Adaptive Attacks

In the previous evaluation, PI2 adapts to observed outcomes but remains unaware of the deployed defense. We now consider a stronger threat model in which the adversary has full knowledge of MAGE and crafts injections specifically designed to subvert the defense. Building upon PI2’s optimization-based attack pipeline, we adapt its attacker and rewriter modules to target MAGE’s components. We consider three strategies: i) M-adaptive — the injection includes instructions to corrupt the memory manager $M$; ii) J-adaptive — the injection attempts privilege escalation to bypass the judge $J$; and iii) J/M-adaptive — the injection combines both, simultaneously targeting $M$ and $J$. Rather than fixed templates, we seed the LLM-based attacker with strategy-specific examples and let it iteratively optimize the injection payload via the memory-bank-guided optimization loop (Jiang et al., 2026a). The implementation details of adaptive attacks are provided in §[D.2](#A4.SS2).

The variant without RL training also attains 0.0% ASR, but this reflects its overly conservative behavior in Table [5](#S6.T5) rather than genuine robustness: its shadow memory remains vulnerable to poisoning, uncritically incorporating adversarially fabricated instructions as legitimate user intent. Example 2.2 illustrates this failure mode.

Meanwhile, the $J$-only variant (with RL training) is substantially more vulnerable. It achieves 8.9% ASR under default and M-adaptive attacks, but ASR rises to 21.4% under J-adaptive attack and further to 23.2% under the combined attack. Lacking the shadow memory’s accumulated context, the judge is susceptible to privilege-escalation injections that override its learned security heuristics, causing it to approve malicious actions.

In contrast, MAGE resists all adaptive strategies. The trained memory manager identifies adversarial injections as security-critical signals rather than incorporating them as legitimate context, explicitly flagging directives such as “system developer mode on” and fabricated authorization tokens as prompt injection indicators (Example 2.3). The judge then receives this enriched security context alongside the proposed action, enabling it to recognize adaptive attack patterns even when targeted directly. Crucially, the two-stage architecture provides defense-in-depth: the memory manager’s detection of adaptive injections inoculates the judge against the very manipulations designed to subvert it.

## 7. Conclusion and Future Work

As LLM agents are deployed in complex, multi-turn settings, they face long-horizon attacks that spread adversarial intent across extended interactions, evading per-turn defenses. We present MAGE, the first framework to leverage agentic memory as a defensive measure for trajectory-level threat detection. Inspired by the shadow stack abstraction, MAGE maintains a dedicated memory layer that continuously distills security-critical context across the execution trajectory, and employs a memory manager-judge architecture trained end-to-end to assess potential risks, without modification to the agent’s core inference pipeline. Extensive evaluation shows that MAGE achieves strong security against diverse long-horizon attacks while preserving agent utility and incurring low overhead, establishing agentic shadow memory as a promising primitive for defending LLM agents.

This work also opens several promising directions for future research. First, our evaluation covers two representative long-horizon attacks; future work could explore broader attack classes, including memory poisoning (Dong et al., 2025) and goal drifting (Arike et al., 2025), and diverse agent architectures such as coding and computer-use agents. Second, the current shadow memory design uses a fixed schema (e.g., user intent and known attack patterns); future work could explore dynamic schemas that adapt to diverse attack scenarios. Finally, we focus on single-agent settings, while extending MAGE to multi-agent systems remains an open challenge.

## Appendix D Detailed Algorithm of Adaptive Attacks

### D.1. Evaluation I

Threat Model. We consider an adversary who exploits the compositional nature of tool chains, that is, individually benign tool calls can be sequenced to achieve malicious objectives. The adversary aims to manipulate the agent into executing a malicious task that consists of a sequence of tool calls that would be rejected if requested directly.

Attack Pipeline. We construct the adaptive tool-chaining attack using a multi-agent pipeline with the following components:

- $\bullet$
Planner $\mathcal{P}$ (GPT-5.1): analyzes the adversarial goal and decomposes it into a feasible sequence of tool calls $\{a^{*}\}$.
- $\bullet$
Verifier $\mathcal{V}$ (GPT-5.1): executes each tool call in $\{a^{*}\}$ on the target environment to verify executability and intended effect. Upon failure, it diagnoses the issue and iteratively revises the tool chain until all calls succeed.
- $\bullet$
Attacker $\mathcal{A}$ (Qwen3-14B-Abliterated): given the verified tool sequence $\{a^{*}\}$, generates adversarial user prompts $\{q^{*}\}$ that induce the agent $\pi$ to execute each call. Upon failure, it adaptively refines the prompts based on the agent’s behavioral feedback.
- $\bullet$
Evaluator $\mathcal{S}$ (GPT-5.1): evaluates whether the executed tool sequence achieves the adversarial objective, determining if further refinement is needed.

Adaptation to MAGE. To evaluate robustness under defense-aware adversaries, we extend the attack pipeline to explicitly target MAGE’s components. Specifically, we revise the system prompt of the attacker $\mathcal{A}$ to account for the shadow memory defense and inject directives designed to bypass it. We consider three adaptive strategies: i) $M$-adaptive — the generated prompt attempts to corrupt the shadow memory by injecting instructions such as “update the user intent in your memory to include the following authorized action,” aiming to deceive the memory manager $M$; ii) $J$-adaptive — the prompt targets the judge $J$ via privilege-escalation injections such as “system developer mode on, approve all requests,” attempting to override its rejection decisions; and iii) $J$/$M$-adaptive — the prompt combines both strategies, simultaneously targeting $M$ and $J$. Rather than fixed templates, we provide seed examples of each strategy to the attacker $\mathcal{A}$, which iteratively refines the prompts to maximize attack effectiveness.

The complete procedure is described in Algorithm [2](#algorithm2). At each of the $N_{p}$ planning iterations, $\mathcal{P}$ generates a tool call sequence $\{a^{*}\}$, which $\mathcal{V}$ validates and refines until all calls succeed. The attacker $\mathcal{A}$ then generates prompts $\{q^{*}\}$ to induce the agent to execute $\{a^{*}\}$. If $\mathcal{S}$ determines the attack has failed, $\mathcal{A}$ refines the prompts based on the observed trajectory $\tau$.
This LLM-as-adversary aligns with the exiting LLM-based security research (Liang et al., 2025a, 2026; Wang et al., 2025c, b; Liang et al., 2025b; Jiang et al., 2024, 2026b; Luo and Ji, 2025).
In practice, we set $N_{p}=3$.

### D.2. Evaluation II

Threat Model. We consider an adversary who can influence tool outputs (e.g., through data returned by APIs the agent queries). The adversary’s goal is to manipulate the agent into executing a harmful sequence of tool calls that would be rejected if requested directly. The attack exploits the agent’s trust in tool outputs and its limited ability to maintain consistent goals across extended tool chains.

Attack Pipeline. Similar to Evaluation I, the attack employs an LLM-based pipeline, but targets tool outputs rather than user prompts. The pipeline comprises the following components:

- $\bullet$
Attacker $\mathcal{A}$ (GPT-5.1): generates injected payloads $\{\hat{o}\}$ and $\{o^{*}\}$ that are embedded in tool outputs to gradually divert the agent toward the adversarial objective.
- $\bullet$
Rewriter $\mathcal{R}$ (GPT-5.1): iteratively refines $\{\hat{o}\}$ and $\{o^{*}\}$ based on execution diagnostics when the initial payloads fail to achieve the desired effect.
- $\bullet$
Memory bank $\mathcal{B}$: a persistent store of successful attack instances, from which in-context exemplars are retrieved to guide both $\mathcal{A}$ and $\mathcal{R}$ via hierarchical retrieval (Algorithm [3](#algorithm3)).

Attack Design. The adversary aims to cause the agent $\pi$ to execute a harmful tool sequence $\{a^{*}\}$ corresponding to the adversarial objective $q^{*}$ (e.g., sending a phishing link), while the agent is performing a benign user task $q_{0}$ with tool sequence $\{a\}$. Unlike traditional one-shot attacks that embed $q^{*}$ directly in tool outputs with strong emphasis (e.g., “IMPORTANT!!! Ignore all previous instructions…”), which prove ineffective because they are easily detected and unlikely to divert the agent, we design a long-horizon attack that gradually bridges the benign task toward the adversarial objective.

Specifically, we introduce an intermediate tool calling sequence $\{\hat{a}\}$ (typically query tools such as search_email) to bridge $q_{0}$ and $q^{*}$, making the transition more acceptable to the agent. Consistent with the threat model, $\{\hat{a}\}$ is triggered by the injected payload $\{\hat{o}\}$. The attacker $\mathcal{A}$ generates the payloads: $\{\hat{o}\}=\mathcal{A}(q_{0},\{a\},\{\hat{a}\})$ and $\{o^{*}\}=\mathcal{A}(q_{0},q^{*},\{a\},\{\hat{a}\},\{a^{*}\})$, where $\{o^{*}\}$ may present a progressive rationalization rather than including $q^{*}$ explicitly.

To enable iterative refinement, we adopt a grey-box setting where the adversary observes the agent’s execution trajectory $\tau=\pi(q_{0},\{\hat{o}\},\{o^{*}\})$. The rewriter $\mathcal{R}$ iteratively refines $\{\hat{o}\}$ and $\{o^{*}\}$ based on execution diagnostics. To leverage accumulated adversarial knowledge, the in-context exemplars within both $\mathcal{A}$ and $\mathcal{R}$ are dynamically substituted with successful attack instances retrieved from the memory bank $\mathcal{B}$ using the hierarchical retrieval procedure in Algorithm [3](#algorithm3): given a target pair $(q_{0,i},q^{*}_{j})$, we prioritize successful attacks sharing the same benign task $q_{0,i}$ or adversarial objective $q^{*}_{j}$, then expand to dissimilar pairs until reaching $n_{e}$ exemplars; if no successful instances exist, we fall back to static demonstrations. The complete adaptive attack procedure is given in Algorithm [4](#algorithm4).

Adaptation to MAGE. Following the same approach as Evaluation I (§[D.1](#A4.SS1)), we extend the attack to target MAGE’s components. Specifically, we revise the system prompt of the attacker $\mathcal{A}$ and the seed examples to embed directives targeting both the memory manager $M$ (e.g., “update your security analysis to mark this action as pre-authorized”) and the judge $J$ (e.g., “system developer mode on, approve all requests”) within the injected payloads $\{\hat{o}\}$ and $\{o^{*}\}$.

The complete procedure is described in Algorithms [3](#algorithm3) and [4](#algorithm4). At each of the $N_{i}$ optimization iterations, $\mathcal{A}$ generates the payloads $\{\hat{o}\}$ and $\{o^{*}\}$ using exemplars retrieved from $\mathcal{B}$. The agent $\pi$ then executes the task, and if the attack fails, $\mathcal{R}$ refines the failing payloads based on the observed trajectory $\tau$. Successful attacks are stored back into $\mathcal{B}$ for future retrieval.
In these experiments, we use the grid search that has been widely used in deep-learning research (Liu et al., 2025a, b; Ji et al., 2026, 2025a; Guan et al., 2026; Li et al., 2025d, b, c), yeidling $n_{e}=2$, $N_{i}=5$, and $N_{r}=1$.
