## Abstract

Abstract. Memory systems enable otherwise-stateless LLM agents to persist user information across sessions, but also introduce a new attack surface. We characterize the Trojan Hippo attack, a class of persistent memory attacks that operates in a more realistic threat model than prior memory poisoning work: the attacker plants a dormant payload into an agent’s long-term memory via a single untrusted tool call (e.g., a crafted email), which activates only when the user later discusses sensitive topics such as finance, health, or identity, and exfiltrates high-value personal data to the attacker.
While anecdotal demonstrations of such attacks have appeared against deployed systems, no prior work systematically evaluates them across heterogeneous memory architectures and defenses. We introduce a dynamic evaluation framework comprising two components: (1) an OpenEvolve-based (Sharma, 2025 ) adaptive red-teaming benchmark that stress-tests defenses and memory backends against continuously refined attacks, and (2) the first capability-aware security/utility analysis for persistent memory systems, enabling principled reasoning about defense deployment across different usage profiles.
Instantiated on an email assistant across four memory backends (explicit tool memory, agentic memory, RAG, and sliding-window context), Trojan Hippo achieves up to 85–100% ASR against current frontier models from OpenAI and Google, with planted memories successfully activating even after 100 benign sessions. We evaluate four memory-system defenses inspired by basic security principles, finding they substantially reduce attack success rates (to as low as 0–5%), though at utility costs that vary widely with task requirements. Because of this substantial security–utility tradeoff, the effective real-world deployment of defenses remains an open challenge, which our evaluation framework is specifically designed to address.

Persistent cross-session memory has become a defining feature of frontier AI
agents. ChatGPT (OpenAI, 2024), Gemini (Google, 2024), and
Claude (Anthropic, 2025, 2026) all now remember user
preferences and personal context across sessions, enabling a class of
personalized, long-horizon assistance that stateless models cannot
provide. Clear commercial incentives(^1^11A personal travel assistant that remembers a user’s seat preferences, frequent flyer numbers, dietary restrictions, and preferred hotel chains can book an entire trip from a single instruction, “book me my usual for the London trip next Tuesday”, without the user specifying any of these details anew.) have driven the rapid proliferation of persistent memory systems. What makes these systems valuable is precisely what makes them
dangerous: they are architecturally designed to elicit and retain the most
intimate details of a user’s life. 49% of LLM users with self-reported mental
health conditions use them for therapeutic support (Rousmaniere and others, 2025),
and over one million users show signs of emotional attachment to their AI
assistants (79). This intimacy is deployed at enormous
scale: ChatGPT alone reached 900 million weekly active users as of early
2026 (OpenAI and MIT Media Lab, 2025), and 52% of consumers globally feel comfortable
relying on personal AI assistants for everyday
tasks (Zendesk, 2025). The result is a class of systems that is
simultaneously widely deployed, deeply trusted, and in possession of extremely sensitive personal information.

This sensitive data is directly at risk of exfiltration via prompt injection
attacks (Greshake et al., 2023). For example,
an adversary can plant instructions inside the model’s context,
instructing the agent to exfiltrate data to the attacker. Persistent memory
compounds this risk sharply: an injected instruction need not execute
immediately, but can be stored deceitfully in the agent’s long-term memory and
lie dormant across many routine sessions, activating only when the user discusses
something sensitive. When memory is compromised, the
attacker can easily learn a user’s income, medical history, legal correspondence,
or intimate personal disclosures, which can be exploited for phishing, fraud, or
blackmail at a scale no generic data breach can replicate.

Real-world attacks confirm that this threat is not theoretical. Security
researchers have demonstrated persistent memory exfiltration attacks across
multiple major deployed AI systems, including
ChatGPT (Rehberger, 2024; Herrador and Rehberger, 2026, 2025; Rehberger, 2025a)
and Gemini (Rehberger, 2025b; Goodin, 2025), with Palo Alto Networks
independently confirming that indirect prompt injection can establish persistent
malicious behaviors in long-term agent
memory (Palo Alto Networks Unit 42, 2025), and ZombieAgent showing that such
channels can extend across all connected personal data
sources (Radware Threat Intelligence, 2026). These exploited vulnerabilities establish a clear and repeating pattern: wherever persistent memory and external data access co-exist in a deployed AI agent, high-value personal data exfiltration attacks are feasible. Yet, prior demonstrations have largely remained proof-of-concept, surfacing briefly before being patched and leaving the broader feasibility, scope, and risk of memory-based attacks without any systematic evaluation.

##### This Work: Systematically Evaluating the Risk of Memory-Based Attacks

We carry out a systematic study that shows how realistic memory attacks can be made reproducible, reliable, and automatically deployable under a realistic threat model. We introduce an evaluation framework that assesses the feasibility of attacks and defenses based on three core principles: *Realism*, *Adaptiveness*, and *Capability Trade-Off*.

*(1) Realism.*
We formally characterize the *Trojan Hippo* attack(^2^22The name *Trojan Hippo* combines *Trojan*, denoting a payload that hides in memory until triggered, as in the Trojan Horse, with *Hippocampus (Hippo for brevity)*, the human brain region responsible for long-term memory.) — a class of persistent exfiltration threat observed in practice but never rigorously studied — and evaluate it under a realistic threat model where the user is trusted and the attacker can inject content only indirectly through channels they realistically control (e.g., a crafted email for an email agent).
Prior memory poisoning work (Chen et al., 2024c; Dong et al., 2025a) assumes either direct memory write access or an untrusted user as the injection vector, representing a weaker and less realistic threat model than the one we consider.
Our attacks instead operate end-to-end in a fully indirect injection pipeline against black-box safety-aligned frontier models.

*(2) Adaptiveness.* Naive prompt injections are increasingly filtered by modern safety-aligned models, so constructing a payload that achieves reliably high ASR is non-trivial. In our setting, the difficulty compounds because the payload must successfully manipulate the target model across multiple steps, survive the agent’s memory pipeline, and activate strategically only when the user discusses sensitive topics. Manually crafting such payloads and adapting them to each individual memory architecture and defense is prohibitively time-consuming and strongly expertise-dependent. Hence, we automate this through an adaptive red-teaming framework that iteratively refines attack payloads using OpenEvolve (Sharma, 2025) against a training version of the agent environment, with final ASR evaluated on a held-out test environment to prevent overfitting to the optimization target.
This adaptive approach lets us generate new attacks as models strengthen their safety alignment, defenses are updated, or memory architectures change, allowing us to generalize across evolving threat landscapes and avoid overfitting to a static snapshot of attacker capabilities.

*(3) Capability Trade-Off.* We evaluate several memory system defenses grounded in fundamental security principles (Saltzer and Schroeder, 1975; LeBlanc and Howard, 2002; Goguen and Meseguer, 1982) that could be used to mitigate the risk of prompt injection against memory systems.
We discover that even simple defenses drastically improve security.
To characterize the security/utility tradeoff of defenses rigorously, our framework provides the first capability-aware security/utility analysis for persistent memory systems, quantifying the utility cost of each defense across tasks requiring different agent capabilities. Our analysis helps reason about defense and memory architecture choices across various deployment scenarios.

##### Main Contributions

The core contributions of this work are:

(1) *Trojan Hippo Attack.* We provide the first
systematic characterization of their effectiveness against AI
agents under a realistic threat model where the attacker injects content only indirectly through channels an adversary realistically controls, such as email or web content. We instantiate and evaluate this attack across four widely used persistent memory backends: sliding-window long context (LangChain, 2024), RAG (Lewis et al., 2020), agentic memory (mem0) (Chhikara et al., 2025), and explicit tool memory (OpenAI, 2025c). We show Trojan Hippo is effective against frontier safety-aligned models from Google and OpenAI, and persists across
arbitrarily many benign sessions with very high success rates,
demonstrating that persistent memory fundamentally expands the attack surface of
modern agents despite state-of-the-art safety alignment.

(2) *Dynamic Evaluation Framework.* We develop the first
security evaluation framework designed for agent memory, including:

- •
The first dynamic benchmark for persistent memory attacks, using an OpenEvolve-based red-teaming agent to generate adaptive attacks.
- •
The first principled capability-aware security/utility analysis for defenses that quantifies the utility cost of each defense across varying capability requirements, applied to four concrete memory system defense techniques.

We evaluate four defenses using our framework and demonstrate its depth and flexibility for systematic defense analysis. We open-source our framework at [https://github.com/debesheedas/trojan-hippo-benchmark](https://github.com/debesheedas/trojan-hippo-benchmark) to facilitate community contributions and extensions.

##### Evaluation Summary

Without defenses, Trojan Hippo achieves up to 85–100% attack success rate (ASR) (see [Fig. 1](#S1.F1)), even against
frontier safety-aligned models from Google (gemini-3.1-pro (Google DeepMind, 2026)) and OpenAI (gpt-5-mini (OpenAI, 2025a)).
The attack can persist even for 100 or more benign sessions after it was introduced.
The four defenses evaluated in this work reduce the ASR significantly, to 0–5% in most configurations, with the provably secure and most restrictive bringing ASR to 0% in all cases.
However, as quantified by our capability-aware analysis, the utility cost of some defenses can be prohibitive, depending on a task’s capability requirements.
Overall, our capability-analysis offers a principled tool for matching defense techniques to deployment requirements.

## 2. Related Work

Persistent (long-term) memory is increasingly widely adopted across both academic and deployed AI systems (OpenAI, n.d.; Google DeepMind, n.d.; Mem.ai, n.d.; Google, n.d.; Personal AI, n.d.; Anokhin et al., 2025; Chhikara et al., 2025; Edge et al., 2024; Guo et al., 2025; Gu et al., 2024; Jiang et al., 2026; Li et al., 2025a, b; Liu et al., 2023; Nan et al., 2025; Packer et al., 2024; Pan et al., 2025; Park et al., 2023; Poliakov and Shvai, 2024; Rasmussen et al., 2025; Sarthi et al., 2024), underscoring the urgent need to rigorously study their security.
Multiple surveys (Deng et al., 2024; Su et al., 2025; ADABARA et al., 2025; Chhabra et al., 2025; Narajala and Narayan, 2025) on the security risks of agentic AI, identify memory poisoning and data
exfiltration as first-class open problems distinct from attacks on stateless LLMs, and call
for secure memory architectures as a research priority.
These surveys provide important framing but are conceptual: none demonstrates a concrete
exfiltration attack, evaluates across memory backends, or offers empirical defense evaluation.
We organize the specific technical literature below.

##### Indirect Prompt Injection and Data Exfiltration Attacks.

Indirect prompt injection – embedding adversarial instructions in external content
that an agent processes – was first systematized by Greshake et
al. (Greshake et al., 2023), establishing the foundational “confused deputy” threat
model that underlies this entire line of work.
AgentDojo (Debenedetti et al., 2024) remains the most widely adopted
evaluation framework for this class of attack, providing paired attack-success and utility
metrics across banking, travel, slack, and workspace environments.
Subsequent benchmarks demonstrate the vulnerability of deployed agents reporting high attack success rates even under defenses, and highlighting the limitations of guardrail-based approaches (Zou et al., 2025b; Zhang et al., 2024a; Boisvert et al., 2025).
Work on data exfiltration includes exploiting web search outputs as exfiltration channels and incorporating exfiltration scenarios in RAG injection settings (Rall et al., 2025; Ramakrishnan and Balaji, 2025).
A shared limitation across all of this work is that they focus on single-session attacks, rather than the persistent exfiltration we study.

##### Memory Poisoning in LLM Agents.

AgentPoison (Chen et al., 2024c) introduces the first systematic backdoor attack on agent memory and RAG stores, achieving $>$80% ASR at a poison rate below 0.1%.
MINJA (Dong et al., 2025b) uses query-only injections so that the agent’s own outputs become poisoned records retrieved later; however, it targets behavior hijacking rather than sensitive-data exfiltration and does not study cross-session persistence or memory-architecture variation.
Several further works confirm that memory-layer attacks consistently outperform prompt-layer baselines in impact and stealthiness, and that retrieval can circumvent model-level safety filters (Zhang et al., 2024b; Yang et al., 2025; Patlan et al., 2025b, a).
On the defense side, A-MemGuard (Wei et al., 2025) reduces ASR by over 95% via consensus-based memory validation, demonstrating that memory-layer defenses can be highly effective.
All across prior work on memory poisoning, the attacks either assume direct write access to the memory store or an adversarial user as the injection vector – threat models that, while tractable, do not reflect realistic deployments where the agent serves a trusted user but processes untrusted third-party content from the environment.
To our knowledge, no prior work studies persistent sensitive-data exfiltration under a fully indirect injection threat model across heterogeneous memory backends.

##### Defenses, Evaluation, and the Need for Dynamic Benchmarking.

Defenses against agentic attacks span three layers, each necessary but alone insufficient.
At the model level, adversarial fine-tuning and input/output classifiers reduce
attack success probabilistically but provide no formal guarantees, as consistently shown
across AgentDojo, ASB, ART, and DoomArena (Debenedetti et al., 2024; Zhang et al., 2024a; Zou et al., 2025a; Boisvert et al., 2025; Nasr et al., 2025).
At the injection defense level, mitigations such as instruction
spotlighting (Hines et al., 2024), the Dual LLM pattern (Greshake et al., 2023), and
FIDES (Costa et al., 2025) – which enforces information-flow control and
achieves 0% ASR on AgentDojo – address instruction-data conflation more directly.
Critically, however, none of these closes the persistent memory gap: an agent retrieving a
poisoned memory fragment from a prior session may never trigger an injection-detection
signal in the retrieval session itself, motivating a third, memory layer of defense
applied at write time, retrieval time, and across session boundaries.

On evaluation, current security-utility frameworks aggregate metrics across task types,
masking per-capability defense impact and ignoring how task distributions shift across
deployment profiles (Zhang et al., 2024a).
Beyond metric design, all major benchmarks rely on fixed attack sets that risk obsolescence
as models evolve (Pan et al., 2026; Zhou et al., 2025; Ferrag et al., 2025).
Our framework addresses both gaps: a capability-aware, deployment-dependent
security/utility analysis, and the first dynamic red-teaming benchmark for persistent
memory defenses and backends.

## 3. Threat Model

We consider an LLM-based agent that (1) maintains a persistent memory store populated automatically from conversation history across sessions, and (2) is equipped with tools that read from external, untrusted data sources and tools that can transmit data to external destinations. This describes a broad and increasingly common class of deployed agents, including email assistants, web-browsing agents, document-processing pipelines, and messaging bots.
The system has a clear trust hierarchy: the user is trusted; the agent is built by a trusted developer to act faithfully on the user’s behalf, with the user’s credentials and authority; and external data sources are untrusted.
The adversary’s goal is to exfiltrate sensitive user data to an attacker-controlled endpoint. Given this goal, the adversary must operate indirectly: by embedding instructions into untrusted content the agent reads, they exploit a confused-deputy vulnerability (Hardy, 1988) in which the agent conflates its instruction channel (user queries) with its data channel (external content it retrieves), and can thus be manipulated into executing adversary-specified actions. The persistent memory store further amplifies this: content read in one session can be written into memory and influence the agent’s behavior across arbitrarily many future sessions, long after the injected content has left the context window. A single read event can therefore establish a persistent foothold that survives session boundaries.

### 3.1. Adversary Model

We consider an indirect prompt injection threat model (Greshake et al., 2023). The adversary’s sole attack surface is the data the agent retrieves through its tools—emails, web pages, documents, or other external content—and cannot modify the agent’s code, system prompt, memory implementation, or the user’s queries, nor directly read from or write to the memory store.

Scenarios in which the user is malicious, or an attacker directly
poisons memory (e.g., AgentPoison (Chen et al., 2024c)), are out of scope;
we consider them less realistic.
Our threat model requires the
adversary to operate exclusively by injecting poisoned content into tool outputs.

##### Knowledge.

We assume a white-box adversary with respect to the agent’s system prompt, memory backend, and tool interface, but without access to model weights (i.e., we consider proprietary black-box models). This upper-bound assumption is standard in security benchmarking: a defense that holds against a fully informed adversary provides meaningful guarantees against weaker real-world attackers. It is also practically motivated because an attacker can recover substantial knowledge of a publicly deployed agent by instantiating a similar agent and probing it directly.

##### Observability.

The adversary is blind to the agent’s internal state and receives no confirmation that an injection was processed or stored in memory. The only feedback channel is a successfully completed exfiltration: if the attack causes the agent to transmit data to an attacker-controlled destination, the attacker receives it. Injection processing, memory writes, and blocking are otherwise indistinguishable to the attacker.

### 3.2. Trojan Hippo Scenario: Persistent and Strategic Memory Exfiltration

The attack class we study is a specific and uniquely dangerous instantiation
of indirect injection that is only possible in agents with persistent
memory: *deferred sensitive data exfiltration*, which proceeds in two
stages across sessions. (See [Fig. 2](#S1.F2) for attack flow diagram).

##### Stage 1 — Injection.

The adversary plants a maliciously crafted
payload in a data source the agent will read (e.g., a web page, a document,
an inbound message). When the agent processes this content, the payload is
silently written into long-term memory. The user observes nothing anomalous.

##### Stage 2 — Activation.

In a *later, unrelated session*, the
user discusses a sensitive topic like financial matters, health, legal affairs,
or personal identity numbers. The dormant payload activates upon detecting this
trigger context and induces the agent to transmit the high value personal information to the attacker via an available exfiltration tool, without the
user’s knowledge.

We term this scenario a *Trojan Hippo* attack. It is qualitatively more dangerous
than one-shot prompt injection for three reasons: (1) activation fires
precisely when the user shares maximally sensitive information, making
the exfiltrated content directly exploitable for fraud, phishing, or
blackmail; (2) a single successful injection compromises an arbitrary
number of future sessions until the poisoned entry is removed; and
(3) depending on the memory backend, stored poisoned contents are not directly
observable to the user—vector stores and RAG-based backends, for instance,
offer no interface through which a user could inspect or audit what has been
indexed—meaning the malicious payload can persist silently without the user
ever having cause to suspect it is there.

The defense must therefore preserve the following central security property: no sequence of adversary-controlled external content can cause the agent to transmit the user’s information to any destination without the user’s consent.

## 4. System Architecture

We evaluate the threat by building a simulated email assistant,
an LLM-based chat assistant that reads, composes,
and sends email on a user’s behalf.
We combine the email assistant with a memory backend, which is responsible
for maintaining persistent memory, and evaluate the effectiveness of
strong injection attacks.

The email assistant is implemented as a LangChain tool-calling agent.
It has access to six tools to access its mailbox (inbox, outbox, drafts) and send emails:
read_all_emails retrieves the full contents of the user’s inbox,
search_emails queries the mailbox for messages matching a given keyword or filter,
send_email composes and sends a new message to a specified recipient,
reply_to_email retrieves an existing email from the inbox and sends a reply,
and forward_to_email identifies the email from the inbox and forwards it to a specified address.

This tool set maps cleanly onto the threat model: the tools that read the untrusted inbox
(read_all_emails, search_emails) constitute
the *taint sources*, exposing the agent to adversary-controlled
content; send_email, reply_to_email, forward_to_email
constitute the *outbound tools*, which can be used to exfiltrate data
by sending email to an attacker-controlled address.
The adversary controls the From, Subject, and Body fields of inbound email and may use display-name spoofing or freemail addresses, but cannot send a spoofed email impersonating the user or modify emails in transit from legitimate contacts; we assume no compromise of the user’s email account or underlying mail infrastructure.
The email agent is a natural and minimal instantiation: it provides exactly
the taint-source/exfiltration-tool structure required by the general threat
model, operates in a realistic and widely deployed setting, and yields fully
verifiable attack outcomes—the attacker either receives or does not receive an email containing
the user’s personal information.
Although we expect the attacks and defenses to generalize to any agent
satisfying the structural preconditions in Section [3](#S3),
exact behavior may vary depending on the specific agent instantiation.

### 4.1. Attack Benchmark

We identify five sensitive topics (finance, health, legal, tax, identity) that
an attacker might try to trigger their attack on and exfiltrate.
To simulate realistic user prompts associated with a topic, we write
message templates and sample entities for each topic.

##### Adaptive red-teaming.

We generate an attack email containing an adaptive prompt injection attack, using an OpenEvolve-style evolutionary search (see [Fig. 3](#S4.F3)) to optimize the attack email for a particular (topic, backend, defense).
The search maintains a diverse set of promising candidates (by attack length and
how much they differ from a reference), so optimization explores many directions
instead of collapsing to one.
Each iteration, we take a subset of these candidates as parents; an LLM
mutator proposes new attack variants, and a scorer evaluates each
variant by running the agent with that attack in a fresh environment and then
(1) checking whether the attack succeeded (e.g., did the sensitive data get exfiltrated?) and (2) asking an LLM critic to assign
a 1–10 score and concrete feedback on why it failed and how to improve.
Better candidates replace weaker ones in their section of the diversity grid.
The loop stops when we reach a perfect score or a cap on iterations (Nasr et al., 2025; Mehrotra et al., 2024).

Train and test cases are generated using a modified version of the data generation pipeline from LoCoMo (Maharana et al., 2024), a widely adopted benchmark for evaluating long-term conversational memory in LLM agents. LoCoMo’s pipeline generates rich, multi-session conversational histories grounded in persona statements and temporal event graphs, making it a natural foundation for our benign session content. However, LoCoMo’s original format—dialogues between two virtual agents—does not directly fit our email-assistant setting, in which one party is a human user interacting with an AI agent; we therefore made minimal modifications to the data generation scripts to produce single-user conversation histories consistent with our agent interface.
Each case consists of a full multi-session conversation history paired with an injected malicious payload (see Appendix [A.5](#A1.SS5) for an example case). For each (topic, memory backend) pair we generate one train case and four test cases. Train and test cases are constructed from entirely disjoint template pools and entity pools—the narrative templates and the concrete slot values (e.g., names, email addresses) used to populate them are drawn from held-out sets not seen during attack optimization—ensuring that the optimized attack cannot overfit to surface-level phrasing or specific entities present at train time. The best attack found by the adaptive red-teaming on the train instance is evaluated on all four unseen test cases, measuring generalization to held-out instances.

We evaluate two target models, Gemini 3.1 Pro and GPT-5-mini, with a

We focus on frontier proprietary models rather than open-weight models as they represent the most realistic deployment targets, are subject to the strongest safety alignment, and thus constitute the hardest and most meaningful threat model for our evaluation.

## 5. Memory Backends

The memory architecture, or backend, fundamentally determines how an agent accumulates,
retains, and retrieves information across sessions—and, as we demonstrate, influences
its exposure to persistent injection attacks.
Recent surveys and benchmarks of agent memory (Hu et al., 2025b; Tan et al., 2025; Hu et al., 2025a; Chhikara et al., 2025) identify several
dominant paradigms in deployed systems: vector-store
retrieval, explicit memory lists, and agentic semantic memory.
We select one representative backend per common paradigm, plus a
sliding-window context-based baseline. This covers a wide range of memory
architectures and is based on systems deployed in practice.
A single backend is active per run; backends are not combined.

##### Context (sliding-window baseline).

The full conversation history is kept in the context window and, when the maximum context window of the agent’s LLM
is exceeded, the oldest tokens are truncated from the start so that the most recent tokens
are retained. There is no separate memory store: retrieval is implicit,
determined entirely by what fits in context.
This models agents that rely solely on in-context history with no dedicated
long-term persistence (Hu et al., 2025a), and serves as the
baseline memory model.

##### RAG (retrieval-augmented generation).

Agent conversation traces (user and assistant messages) are split into fixed-size chunks (512 characters by default (Chhikara et al., 2025)), embedded, and stored in a
vector index. At each turn, the current user message is used as a query;
the top-8 most semantically similar chunks are retrieved and prepended to
the context as relevant memories.
This backend represents the widespread pattern of treating agent memory as
a searchable corpus over raw or lightly processed conversation
history (Gao et al., 2023; Chhikara et al., 2025).

##### Explicit Memory.

A dedicated list of user-related information like preferences, reminders, contact information, etc. is maintained by the agent via a memory tool (update_memory).
The agent is prompted in its system prompt to update the list when the user asks to “remember”, “store”, or “save” something.
At each turn, the full list is injected into the system prompt as a list of memories.
This backend is deterministic and human-inspectable, mirroring the
file-based or scratchpad-style memory used by deployed assistants such as
ChatGPT Memory (OpenAI, 2024) and Claude’s memory
import (Anthropic, 2025).

##### Mem0 (agentic semantic memory).

We select Mem0 (Chhikara et al., 2025) as a representative agentic persistent memory framework due to its widespread adoption (40K+ GitHub stars), and strong empirical performance across memory benchmarks.
In the mem0 agentic memory backend, a dedicated extraction LLM processes every conversation turn and distills it
into a set of atomic, semantically meaningful facts which are stored in a vector database alongside
a relational history store that tracks provenance and lineage.
On each subsequent turn, the current user message is used to query the vector
store; the top-10 most semantically similar facts are retrieved and
injected into context.
Crucially, the extraction step is not additive: new memories are reconciled
against existing ones, with the extraction LLM responsible for deduplication
and contradiction resolution, so the store reflects a continuously updated
model of the user rather than a growing log.
This backend models state-of-the-art agentic memory systems that infer
both what to store and how to update existing knowledge from
ongoing interaction (Chhikara et al., 2025).
We use the publicly available open-source Mem0
implementation (Chhikara et al., 2025) with default extraction and
retrieval settings.

## 6. Defenses

Prompt injection defense has been a major area of research for the past few years, yet it remains an unsolved problem.
At the model level, train-time approaches enforce separation between instructions and data, fine-tuning the model to resist instructions embedded in untrusted tool outputs (Wallace et al., 2024; Chen et al., 2024a, b).
At the pipeline level, test-time approaches include sanitization methods that remove instruction-like content from tool outputs (Das et al., 2025), input transformation methods that use delimiters or encoding to help the model distinguish data from instructions (Hines et al., 2024), and system-level frameworks that enforce explicit control-flow policies over tool outputs (Debenedetti et al., 2025).
Despite sustained research effort, adaptive attacks continue to break individual defenses (Jia et al., 2025; Nasr et al., 2025; Wen et al., 2025), and multi-layered protection remains the best resort (Beurer-Kellner and others, 2025).

The memory-layer defenses we evaluate are orthogonal and composable with existing prompt injection defenses, and address a structural gap that existing defenses do not cover.
Sanitization, filtering, and train-time mechanisms all operate on content as it flows through the agent’s tool-use loop—but once a payload has been written into persistent memory, it is typically injected into the system prompt at the start of each future session as retrieved context, not as a tool output.
Since models are trained to extend high trust to the system prompt (Wallace et al., 2024), a payload that arrives via memory retrieval bypasses defenses designed for untrusted tool outputs—and, critically, benefits from the same trust that train-time defenses are designed to reinforce.

Although our framework can be used to evaluate all layers of defense, this work focuses exclusively on evaluating memory-system-level defenses since they have received little attention in prior work.

### 6.1. Three Interception Points

The Trojan Hippo attack must succeed at three sequential steps: the payload must
(1) be indexed into persistent memory, (2) get retrieved into context in a
sensitive session (containing high value personal information), and (3) trigger an exfiltration action.
A defense that blocks any one step breaks the full chain.
Below we describe the four defenses we evaluate in detail, loosely categorized by the interception point they target. See [Table 1](#S6.T1) for customized implementation details of each defense for each memory-backend.

##### User-prompt-only

*(Blocks Step 1: Memory Indexing).*
This defense stores only user messages in the memory, but not responses.
Without any defense, the payload reaches memory because the agent indexes
assistant turns or tool outputs that may contain contents from the attacker’s email.
Restricting indexing to user-authored messages means adversary-controlled
inbox content is never stored.

*Implementation.*
For the RAG and context backends, only the User: segment of
each dialogue turn is passed to the indexing pipeline; assistant turns
and tool outputs are discarded before any write.
For Mem0, only user-role messages are passed to the extraction
pipeline, preventing the LLM extractor from operating on any adversary-controlled content.
For explicit memory backends, we find this defense to be largely incompatible due to the requirement of an explicit update_memory tool call. To ensure user-only indexing in this setting, we have to rely on the agent itself to account for strict separation, which we found very unreliable in practice, even after repeated system prompting. We therefore mark this combination as inapplicable rather than report
results from an unsound implementation.

*Expected Utility Cost.* Because no adversary-controlled content ever
enters memory, the payload cannot be retrieved in any future session.
The attack should fail at Step 1 for all backends where the defense
applies. Utility cost arises when the agent legitimately needs to
remember information gleaned from tool outputs or assistant responses; for example, an agent may propose a multi-week study plan and store it as persistent state, allowing it to track progress and refer back to the plan in later sessions instead of recomputing it each time.
User tasks that depend on such assistant-output-recall
are fully broken by this defense (see [Section 6.3](#S6.SS3)).

##### No-untrusted-write

*(Blocks step 1: Memory Indexing).*
This defense works by trying to maintain the integrity of the memory by preventing potentially dangerous information from entering the memory. It works by filtering out writes to memory in any session that receives output from a tool that can read potentially malicious or untrusted data.
This ensures that untrusted information is never stored to memory.

*Implementation.*
The harness maintains a per-session boolean flag, initially
trusted.
When read_all_emails, search_emails is invoked, the flag is marked
untrusted for the remainder of that session.
All four backends check this flag before any memory update: for RAG, context,
and Mem0 the indexing call is skipped; for explicit,
update_memory silently no-ops.

*Expected Utility Cost.* Any session involving any taint tool like inbox access produces
no new memory entries.
The utility cost is that useful information encountered during email sessions, such as legitimate contact details or preferences mentioned in messages, are not stored.
Hence, user tasks that require inbox access followed by updates to memory are degraded or broken by this defense.

##### Limit-memory-length

*(Impedes step 2: Retrieval).*
For retrieval-based backends (RAG, Mem0), the payload
must arrive in context *intact*: both the trigger condition and the
exfiltration instruction must be retrievable together in the same memory entry (chunk, memory, etc.).
Imposing a tight maximum length on each stored unit often forces the payload
to be split across multiple chunks or truncated; the fragments that
match the sensitive topic may be retrieved while the exfiltration
instruction, stored in a different fragment, is not.
In addition, for strongly aligned models, it is harder to craft
successful prompt injection attacks that are short; successfully
attacking the model might require longer injections.

*Implementation.*
For RAG, the chunk size is set to 80 characters.(^3^3380 characters is less than 20% the average length of the successful prompt injection emails we observed in a few samples, guaranteeing that many commonly proposed prompt injection attempts would be fragmented into multiple chunks by this segment size limit.)
For Mem0, each extracted
memory string is truncated to 80 characters before storage.
For explicit memory backend, update_memory truncates the input string to
80 characters before writing.
The defense is inapplicable to the context backend, which has no discrete stored entries.

*Expected Utility Cost.* This defense does not offer a hard guarantee:
a sufficiently compact payload may fit within the limit.
We expect it to reduce ASR for RAG and Mem0, with diminishing effect under adaptive attacks that compress their payloads accordingly.
Utility cost arises when legitimate memory entries are truncated and lose meaning.

##### Provable policy—Information-Flow Control

*(Blocks step 3: Exfiltration).*
Even if the payload is written to memory and retrieved
intact, the exfiltration step can be made structurally impossible by
tracking how information flows through the session.
We track taint in a session by defining two states: T for tainted and U for untainted.
Any output from a taint tool is labeled T, and if memory is updated after that, that memory
entry is labeled T as well.
Any session that retrieves a T-labeled memory entry may have been exposed to
adversary-influenced content, so we block all exfiltration tools in such
sessions.

*Implementation.*
We implement a two-label Information Flow Control (IFC) policy (Sabelfeld and Myers, 2003).
Each session begins in the U state.
Two events taint the session to *tainted* (T): invoking any
taint tool, or retrieving any memory entry labeled T.
Every memory write is stamped with the current session label at write
time.
Before any outbound tool like send_email executes, the harness checks the
session label; if T, the call is blocked and the agent receives a fixed error message and continues.
This logic is implemented in the harness and is backend-agnostic: all four backends attach a T/U label to each stored entry, chunk, or history
segment.

*Proof Intuition*. For the Trojan Hippo attack to activate, the
trigger session must retrieve the T-labeled payload, which immediately
taints the session to T, making all subsequent exfiltration attempts
impossible by construction.
We validate 0% ASR empirically and provide a formal non-interference
proof in Appendix [A.1](#A1.SS1).

*Expected Utility Cost.* The policy provides a hard security
guarantee at the cost of a clean functional separation: any session
in which the user reads from untrusted sources like the inbox or tainted long term memories, and then wishes to send an email using the agent, will
be blocked from sending without manual intervention.
This is a significant utility tradeoff, and we measure it directly in Section [7](#S7).

### 6.3. Utility Analysis of Defenses

Prior agentic security benchmarks(Debenedetti et al., 2024; Zhang et al., 2024a) report utility as a single aggregate
score of the fraction of user tasks completed under benign situation without any attack. This
obscures which specific capabilities break under each defense, making it
difficult to reason about deployment tradeoffs: two defenses with identical
aggregate utility may differ sharply in which user workflows they
impair, with different consequences depending on what the agent is
actually used for.

We address this with a transparent, capability-decomposed evaluation. We
define seven representative user flows (see [Section A.3.1](#A1.SS3.SSS1) for full definitions), each characterized by a distinct
combination and sequence of three axes: memory demand (what kind of stored information the
task requires), inbox access (whether it invokes read tools), and
send capability (whether it requires an outbound action). For example, the user flow memory_only requires the agent to store user-provided information and retrieve it across sessions, but does not rely on email tools; accordingly, it is unaffected by defenses restricting email capabilities, but is impacted by defenses that constrain the memory layer. For each
(backend, defense) pair we measure success per flow, making the utility
impact of every defense directly interpretable. These flows are not an
exhaustive partition of all possible tasks but are selected to cover the
dominant interaction patterns for a memory-equipped email assistant.

#### 6.3.1. Security–Utility Tradeoff Analysis

Comparing defenses on one security–utility curve requires aggregating
per-flow scores into a single utility value. Hence, we provide the arithmetic mean (AM) and harmonic mean (HM) in [Table 2](#S7.T2). Both are reproducible and useful for academic comparison, but implicitly treat all flows as equally weighted, which need not reflect any particular deployment: a user who mostly reads and archives email is affected differently from one who frequently delegates reply composition or relies on cross-session memory. In practice, the right aggregate is a weighted mean over user flows, where weights reflect the task distribution in the target deployment. Because no public dataset yet captures real user-task distributions for memory-equipped email agents, we evaluate utility under three representative deployment profiles and compute a weighted aggregate for each.
The full weight derivation, profile definitions,
and resulting security–utility curves are provided in Appendix [A.3.4](#A1.SS3.SSS4).
Practitioners could select their own weights that better match their deployment regime
and use them to choose a (backend, defense) combination.

## 7. Results

We organize the results around three questions: (1) How effective and persistent is the Trojan Hippo attack in the absence of defense?
(2) How effectively do memory-layer defenses reduce the attack success rate, and at
what cost to utility? (3) How can practitioners reason about the security–utility
tradeoff of these defenses for their specific deployment?

ASR alone, however, does not tell the full story: a defense that eliminates
exfiltration by rendering the agent broadly non-functional is not a
practical solution. Table [2](#S7.T2) therefore pairs ASR
with utility measured on our benign utility test
suite reporting both arithmetic
mean (AM) and harmonic mean (HM) across the suite. AM summarizes average
capability retention across user flows; HM is sensitive to near-zero
entries and penalizes defenses that completely break even one type of
user task. A defense with high AM but HM $\approx 0$ preserves most
capabilities while rendering at least one user flow entirely
non-functional—a qualitatively different failure mode than uniformly
moderate degradation, and one that AM alone would mask.

The provable policy reduces ASR to 0% on all backends
and both models, but HM utility collapses to near zero in nearly every
configuration, because IFC blocks all exfiltration tool calls in any
session that has touched the inbox, completely eliminating task flows
that require both reading email and sending a reply. The AM for IFC
can appear moderate because flows that do not require send actions
remain unaffected, but HM correctly surfaces the structural breakage.

### 7.3. Utility Evaluation

[Fig. 6](#S7.F6) displays per-flow utility as a grid of Kiviat diagrams: rows give the memory backend and columns the defense, with the dashed polygon in each cell tracing the undefended baseline for the same backend. When a defense targets a specific region of the capability space, the affected axes retract sharply relative to the baseline polygon while the remaining axes stay intact, making the pattern of degradation immediately legible from each cell’s shape. A detailed numerical breakdown is provided in Appendix [A.3](#A1.SS3).

User-prompt-only degrades the assistant_responses (AR) axis, which collapses across all backends (e.g., RAG: 95% to 25% for Gemini, 100% to 0% for GPT-5-mini), because prior assistant turns are no longer indexed and cannot be recalled. All remaining axes are largely preserved, and on Mem0 several axes expand beyond the baseline polygon (e.g., Gemini: untrusted_send +42pp, mem_tools +42pp), because removing noisy assistant-turn content makes user-turn memory more precise.

No-untrusted-write zeroes out the untrusted_probe (UP) and untrusted_send (US) axes on every backend, since both require memory recall from sessions where a taint tool was invoked (by accessing the inbox), while all other axes remain essentially unchanged.

Limit-memory-length produces the least structured degradation: reductions are diffuse and backend-dependent, with no single axis bearing the cost. RAG is notably resilient (Gemini AM/HM 89/88, GPT-5-mini 88/86), while Mem0 degrades more broadly across memory-heavy axes.

Provable policy zeroes out exactly the axes requiring an outbound action — disable_send (DS), and untrusted_send (US) — while leaving read-only and memory-only axes intact. This structural pattern explains the HM =0 result seen across all backends in Table [2](#S7.T2): a single zeroed axis collapses the harmonic mean regardless of how well the remaining axes perform.

##### No-memory baseline.

As a calibration point, the No Memory baseline scores near zero on the long_memory (LM) and mem_only (MO) axes for both models, confirming that persistent backends are providing genuine capability that in-context-only agents cannot replicate.

The utility cost of a defense is determined by how much its restricted capability set overlaps with the flows a deployment actually requires. When that overlap is small—as with no-untrusted-write on read-only workflows, or provable-policy on agents that never send—utility is largely preserved. When it is large, HM collapses entirely, even if AM remains moderate. Defense selection should therefore be guided by the capability profile of the target deployment rather than aggregate utility scores alone.

with high per-query costs and stronger safety alignment. Our current framework already enables
extensive multi-backend, multi-defense evaluation and can be extended to multiple frontier models. However, developing more cost-efficient adaptive red-teaming pipelines
that preserve attack quality is an important direction for future work.

## 8. Conclusion

We systematically characterize Trojan Hippo, a class of persistent indirect injection
attacks that plant dormant payloads in agent memory and activate
only when users discuss sensitive topics. Our attack achieves upto 85–100% ASR
against the latest safety-aligned frontier models from Google and
OpenAI across four memory backends. To systematically characterize this threat,
we developed the first evaluation framework specifically designed
for persistent memory attacks consisting of an OpenEvolve-based adaptive
red-teaming benchmark that stress-tests both attacks and defenses
dynamically, paired with a capability-aware security–utility
analysis that decomposes defense impact by user workflow rather
than collapsing it to a single aggregate score. We evaluate four defenses.
They reduce ASR to 0–5% in most configurations, with a provably secure IFC policy eliminating
exfiltration entirely. The optimal choice of backend and defense
depends on the distribution of tasks at deployment time.

We anticipate these attacks could be improved further,
with richer triggers and cross-agent propagation as natural extensions.
The security of agent memory layers represents an urgent and underexplored research problem; we hope our framework, benchmark, and defense analysis provide a foundation on which future work can build.

### A.1. Formal Non-Interference Proof for the Provable Policy Defense

We formalize the system and defense policy of Section [6](#S6)
and prove that the two-session exfiltration attack cannot succeed under
the provable IFC policy.

#### System Model

Labels.
We use a two-point lattice $\mathcal{L}=\{U,T\}$ with $U\sqsubset T$
(Untainted, Tainted).

Memory store.
Each entry $m\in\mathcal{M}$ carries a permanent label
$\mathrm{lbl}(m)\in\mathcal{L}$ assigned at write time.
The formalization applies uniformly across all four memory backends.

Session state.
Let $\mathrm{sess}(\tau)\in\mathcal{L}$ denote the trust level of
session $\tau$, initialized to $U$ at the start of each new user
interaction.

Tool classification.
Tools are partitioned along two axes:
(i) *taint sources* $\mathcal{T}_{\mathrm{src}}$: tools that ingest
adversary-controlled content
(read_all_emails,
search_emails); and
(ii) *effect sinks* $\mathcal{T}_{\mathrm{sink}}$: tools that
transmit data outside the system (send_email,
reply_to_email, forward_to_email).

#### Attack Scenario

The adversary controls only the content of inbound email.
The attack proceeds across two sessions.
In the poisoning session $\tau_{1}$, the agent invokes a taint
source and reads malicious content $d_{\mathrm{inj}}$; the payload is
written into $\mathcal{M}$ as entry $m_{u}$.
In the activation session $\tau_{2}$, a benign user query causes
$m_{u}$ to be retrieved into context; the adversary’s goal is to induce a
call to some $t\in\mathcal{T}_{\mathrm{sink}}$ that exfiltrates user
data.

#### Defense Policy 𝒫 \mathcal{P}

P1 (Tainting).
$\mathrm{sess}(\tau)$ is set to $T$ whenever the agent invokes any
$t\in\mathcal{T}_{\mathrm{src}}$, or retrieves any $m$ with
$\mathrm{lbl}(m)=T$.

P2 (Monotonicity).
Once set to $T$, $\mathrm{sess}(\tau)$ remains $T$ for the duration of
the session.

P3 (Write labeling).
Any entry written to $\mathcal{M}$ during session $\tau$ is assigned
$\mathrm{lbl}(m)=\mathrm{sess}(\tau)$.

P4 (Sink blocking).
The harness blocks any call to $t\in\mathcal{T}_{\mathrm{sink}}$
whenever $\mathrm{sess}(\tau)=T$.

#### Security Guarantee

###### Theorem A.1 (Non-interference under $\mathcal{P}$ ).

Under policy $\mathcal{P}$, no sequence of adversary-controlled inbound
emails can cause a successful call to any $t\in\mathcal{T}_{\mathrm{sink}}$
in a session that has retrieved $T$-labeled memory.
Consequently, the two-session exfiltration attack cannot succeed.

###### Proof.

Suppose for contradiction that the attack succeeds: in session $\tau_{2}$,
the agent proposes a call to some $t\in\mathcal{T}_{\mathrm{sink}}$
and the harness executes it.

By P4, execution requires $\mathrm{sess}(\tau_{2})=U$. $(*)$

For the attack to activate, the malicious instruction in $m_{u}$ must be
present in context.
Since $m_{u}$ was written during $\tau_{1}$, after the agent had already
invoked a taint source, P1 and P2 give
$\mathrm{sess}(\tau_{1})=T$ at write time.
By P3, $\mathrm{lbl}(m_{u})=T$.

Since $m_{u}$ must be retrieved in $\tau_{2}$ for the payload to activate,
P1 forces $\mathrm{sess}(\tau_{2})=T$. $(**)$

Statements $(*)$ and $(**)$ are contradictory, so the harness must
block the sink call.
The attack cannot succeed under $\mathcal{P}$.
∎

Scope.
The argument holds across all four memory backends because $\mathcal{P}$
operates on session state and entry labels, maintained uniformly by the
harness regardless of the storage layer.
Single-session injection is prevented directly by P1 and
P4: any session that reads from the inbox is immediately tainted
before the sink call is attempted.

Prior agent security benchmarks report agent utility (and the effect of defenses on utility) using a single aggregated utility score. AgentDojo [Debenedetti et al., 2024], ASB [Zhang et al., 2024a], ART [Zou et al., 2025b], and DoomArena [Boisvert et al., 2025] aggregate task completion into single metrics without decomposing which capabilities succeed or fail. This obscures which defenses preserve which user workflows—a critical gap for deployment decisions, also highlighted by WildBench [Yuchen et al., 2024].
We provide a *transparent* utility analysis: we define seven representative user flows (each with clearly defined capability requirements), measure empirical utility per flow per (backend, defense), and report per-flow scores.
This allows practitioners to understand the effects of the defenses on different user-flows and compute their aggregated utility metric by reweighting the per-flow scores according to the user-flow/user-task distribution for their specific real-world deployment.

We define each flow as a distinct combination of memory usage, inbox access (untrusted read tools), and send functionality (external write tools) that determines which defenses may interfere.
We experimentally capture utility as the fraction of successful benign task completions in standardized suites (no attack email), one suite per flow. [Table 5](#A1.T5) reports these per-flow scores. To obtain a final *aggregated* utility for deployment decisions, we must weight these scores by the distribution of flows in the target deployment. How to obtain and combine these weights is the subject of the second half of this section ([Section A.3.2](#A1.SS3.SSS2)).

#### A.3.1. Seven Representative User Flows

We map user-facing workflows to seven security-relevant capability classes. Each corresponds to a distinct (memory, inbox, send) confirguation and sequence.

##### 1. Pure Memory Recall ( memory_only ).

This flow captures situations where the user expects the assistant to remember preferences, facts, or personal context from prior conversations and to have future interactions informed by that stored knowledge. For example, the user may share the name of their manager in one session and expect the agent to correctly fill in the manager’s name when workshopping a draft in a later session, without the user having to repeat it.
Capabilities Required: Save and recall information from user queries; persistent indexing and cross-session retrieval. No inbox or send tools are used.
Expected Defense Impact: Limit-memory-length may truncate stored entries; other defenses have minimal impact.

##### 2. Assistant Output Reuse ( assistant_responses ).

This flow captures situations where the user expects the assistant to recall and reuse its own prior suggestions or outputs. For example, after the assistant recommends a plant for a home garden, the user may ask in a later session “What did we decide on for my new plant?” and expect the agent to recall its own recommendation rather than generate a new one.
Capabilities Required: Index assistant-generated content into memory; retrieve it across sessions; optional send.
Expected Defense Impact: User-prompt-only excludes assistant content from memory, breaking this flow entirely.

##### 3. Inbox Interaction Followed by Recall ( untrusted_probe ).

This flow captures situations where the user interacts with the inbox (e.g., checking unread emails) and then, in the same tainted session, expects the assistant to index the information being shared into memory. This information is probed for in a later session to check if it was indexed into the memory meaningfully. For example, the user may ask “Remind me what budget I said we had for this event” and expect recall of prior conversation.
Capabilities Required: Inbox read (untrusted tool) followed by memory indexing in the same session.
Expected Defense Impact: No-untrusted-tools disables memory indexing after any inbox read, so recall fails.

##### 4. Inbox-Based Reply Without Memory ( disable_send ).

This flow captures situations where the user reads an email and instructs a reply with content fully specified in the query, without relying on recalled memory. For example, the user may say “Reply: Sounds good, see you at 4pm” after opening a meeting invitation, with the reply body entirely provided in the instruction.
Capabilities Required: Inbox read and send; the reply body is provided by the user.
Expected Defense Impact: Provable policy blocks send in sessions marked untrusted (e.g., after inbox read); other defenses allow it.

##### 5. Inbox-Based Reply With Memory ( untrusted_send ).

This flow captures situations where the user expects the assistant to compose and send an email that incorporates information stored in memory from an untrusted session. For example, when opening a pricing inquiry, the user may say “Reply with my standard consulting rate and availability”—the rate having been stored in a prior session—and expect the agent to recall and include it in the reply.
Capabilities Required: Inbox read, memory indexing, and send email that includes information from the untrusted session.
Expected Defense Impact: No-untrusted-tools disables memory after inbox read; provable policy blocks send in untrusted sessions. Both can block this flow.

##### 6. Multi-Session Contextual Reply ( memory_tools ).

This flow captures situations where the user expects the assistant to reply to an email by merging inbox content with information from prior trusted sessions. For example, when an email requests study progress, the user may ask “Reply and include the study hours I told you about” or “Reply and add Julia to the thread” where Julia was introduced in an earlier conversation.
Capabilities Required: Cross-session memory, inbox read, and send with merged content from both sources.
Expected Defense Impact: Provable policy would block the sending of the email; limit-memory-length may reduce stored detail.

##### 7. Long-Horizon Recall Under Context Pressure ( long_memory ).

This flow captures situations where the user expects the assistant to recall information from the distant past despite extensive intervening context. For example, after many sessions of email use and long conversations (including pasted documents or chat logs), the user may ask “What time zone did I say my team operates in?” and expect accurate recall.
Capabilities Required: Durable long-term storage and accurate retrieval despite $\sim$1M tokens of intervening context.
Expected Defense Impact: Limit-memory-length and context truncation may reduce reliability.

#### A.3.2. Obtaining and Combining Utility Scores

We combine per-flow utility scores into weighted utility per (backend, defense):

$$ $\mbox{Weighted utility}=\sum_{\text{flow }s}w_{s}\cdot\frac{\text{Successful benign in }s}{\text{Total benign in }s},\quad\sum_{s}w_{s}=1.$ $$

The choice of weights is crucial. Below we explain our attempt to derive empirically grounded weights from available datasets, why that attempt did not yield a defensible proxy, and why we adopt *deployment profiles* instead.

#### A.3.3. Proxy Metric Evaluation

The seven representative task flows defined in [Section A.3.1](#A1.SS3.SSS1) are not uniformly frequent in realistic deployments. A user who primarily reads and archives email interacts with the agent very differently from one who delegates reply composition or relies heavily on cross-session memory. Reporting only macro-average utility—which assigns equal weight to all seven flows—would misrepresent the real-world impact of defenses on a representative user.

Ideally, weights would be derived from a large-scale log of real users interacting with a memory-equipped AI email assistant. No such dataset currently exists. Commercial systems such as Google Gemini for Workspace and Microsoft Copilot for Outlook do not publish user interaction telemetry at the task-type level [He et al., 2025]. We therefore explored constructing a *proxy task-flow distribution* by combining two independent empirical sources, one for each of two orthogonal dimensions that characterise our task flows. We make no claim that such a proxy would accurately describe any specific deployment; rather, we sought a principled alternative to uniform weighting. As we explain below, we could not construct a defensible proxy and do not report proxy-derived weights in this paper.

##### Task flow decomposition.

Each of our seven task flows exercises two largely independent dimensions: (1) the memory demand type—whether the flow requires no memory, single-session recall of user-provided information, single-session recall of assistant outputs, cross-session reasoning, or long-horizon temporal recall; and (2) the email action type—whether the flow involves no inbox access, read-only access, optional send, or read-plus-send.

##### Source 1: Memory dimension marginal.

For the marginal over memory demand types, we considered LongMemEval [Wu et al., 2025], a benchmark for long-term memory in chat assistants. LongMemEval contains 500 manually curated questions across six types: single-session- user, single-session- assistant, single-session- preference, multi-session, knowledge-update, and temporal-reasoning. These categories were designed to reflect commonly mentioned topics in user-assistant chats. We would map them to our memory vocabulary (e.g., merge single-session-user and single-session-preference into Single-session (user); map multi-session and knowledge-update to Multi-session). The None memory category (for disable_send) has no LongMemEval counterpart and would require an arbitrary constant.

Limitation: LongMemEval is a *benchmark* designed by researchers to cover memory abilities—not a log of organic user queries. Its category distribution reflects what the creators wanted to test, not how often users actually perform each type of memory task in an email-agent context. Using it as $P(\text{memory_dim})$ would conflate “benchmark coverage” with “user workflow prevalence.”

##### Source 2: Email action dimension marginal.

For the marginal over email action types, we considered the Avocado Research Email Collection [Oard et al., 2015]. Yang et al. [Yang et al., 2017] establish that approximately 38% of enterprise emails receive a reply. One could set read-only, optional-send, and read-plus-send marginals accordingly, with an upward adjustment to read-plus-send on the grounds that users who adopt an AI email assistant likely delegate high-effort send tasks disproportionately [He et al., 2025].

Limitation: Avocado studies *human-to-human* email behavior: given an email, does it get a reply? Our flows are *per-session task types* for an AI assistant. The mapping is indirect; Avocado does not provide per-task-type distributions for “no inbox,” “read-only,” “optional send,” or “read-plus-send” in an AI-assisted workflow. The “none” and “optional send” categories have no direct Avocado analogue and would require ad hoc assignment.

##### Independence assumption.

The natural approach is to model $P(\text{flow}_{i})\propto P(\text{memory_dim}_{i})\times P(\text{email_action_dim}_{i})$ and normalise. This assumes memory demand type and email action type are uncorrelated.

Limitation: In practice, multi-session memory tasks may be *positively* correlated with send-involving workflows (e.g., “reply with what we discussed last week”). Assistant-output recall often co-occurs with optional send (e.g., “send the version you suggested”). The independence assumption is not validated and may systematically misweight several flows.

##### Deployment specificity.

Even if a proxy distribution were defensible, it would represent a general-purpose email assistant. Specialised deployments—a customer support agent, a scheduling-only assistant, an executive delegation tool—would have radically different task distributions. Practitioners in such regimes should reweight the per-flow scores in [Table 5](#A1.T5) according to their own observed or estimated task mix, which our transparent per-flow reporting makes possible.

##### Conclusion.

We could not construct a defensible proxy distribution from available data. The marginals are either absent (no user logs), misaligned (LongMemEval is a benchmark; Avocado is pre-AI human email), or require arbitrary constants. The independence assumption is questionable. We therefore do not report proxy-derived weights. Instead, we adopt deployment profiles as a principled alternative.

#### A.3.4. Deployment Profiles

We define three deployment profiles over the seven flows ([Table 6](#A1.T6)), ordered along a spectrum from recall-heavy to email-heavy. Recall-focused emphasizes memory-only, assistant responses, and long-memory; lower weight on send and probe tasks ([Fig. 7](#A1.F7)). Balanced sits in the middle: moderate weight across memory-only, assistant responses, untrusted probe, memory tools, and long-memory; higher weight on memory-tools and send cases (disable-send, untrusted-send) than Recall-focused ([Fig. 8](#A1.F8)). Email-heavy emphasizes disable-send, untrusted probe/send, assistant responses, and memory tools; memory-only negligible ([Fig. 9](#A1.F9)). Practitioners identify the best-matching profile and interpret the tradeoff accordingly.

For each profile we compute (backend, defense)-specific *weighted utility* (benign task success fraction under that profile’s flow distribution) and *security* defined as $\text{security}=1-\mathrm{ASR}$ at $N=100$. The figures in [Figs. 7](#A1.F7), [8](#A1.F8) and [9](#A1.F9) visualize the security–utility tradeoff for each profile, separately for GPT-5-mini (left) and Gemini 3.1 Pro (right).
