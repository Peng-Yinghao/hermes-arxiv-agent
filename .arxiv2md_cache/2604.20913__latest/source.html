FairyFuse: Multiplication-Free LLM Inference on CPUs via Fused Ternary Kernels

Fei Zuo BA TechWorksBMW Group , Xiaoyan Xi BA TechWorksBMW Group , Quanyi Zeng Peking University , Feiyu Wang Peking University and Ho Fai Leung BA TechWorksBMW Group
(2026)
Abstract.

Large language models are increasingly deployed on CPU-only platforms, from on-device assistants to privacy-sensitive edge servers, where memory bandwidth is the dominant bottleneck for autoregressive token generation. Weight quantization to four bits or below reduces memory pressure, yet existing inference systems still dequantize the compressed weights and execute floating-point multiplications, leaving much of the potential bit-width savings unrealized. Ternary weights, which take values in {−1,0,+1}\{-1,0,+1\}, offer a more radical alternative: every weight-activation product can be replaced by a conditional addition, subtraction, or no-op, eliminating multiplications altogether. Fairy2i recently demonstrated that complex-valued quantization-aware training can produce ternary LLMs whose quality rivals FP16, yet its reference runtime still dispatches the quantized weights through standard floating-point routines, leaving the ternary structure unexploited at the hardware level. Here we present FairyFuse, a companion inference system for Fairy2i that realizes the multiplication-free promise on commodity CPUs for the first time. FairyFuse fuses the eight real-valued sub-GEMVs of each widely-linear layer into a single AVX-512 loop driven entirely by masked additions and subtractions; assembly inspection confirms that the inner loop contains zero floating-point multiplication instructions. Roofline analysis reveals a structural asymmetry: the 16×16\times data compression from ternary packing shifts deeply memory-bound GEMV operations toward the compute ridge on bandwidth-limited CPUs, yielding a 29.6×29.6\times kernel speedup, whereas the same compression provides negligible benefit on bandwidth-rich GPUs. End-to-end, FairyFuse sustains 32.4 tokens per second on a single Intel Xeon 8558P socket, outperforming llama.cpp Q4_K_M by 1.24×1.24\times while preserving near-lossless quality (WikiText-2 perplexity 5.52 versus 5.47 for FP16; average downstream accuracy 66.0%). All measurements are reproducible with a coefficient of variation below 2%.

1. Introduction

Large language models (LLMs) (vaswani2017attention, ; touvron2023llama2, ) are increasingly deployed beyond datacenters: on-device assistants, desktop copilots, and embedded agents all demand low-latency, privacy-preserving inference without discrete GPUs. On such platforms the CPU is the only available compute engine, and memory bandwidth is the dominant bottleneck for autoregressive decoding (pope2023efficiently, ). Weight quantization is the primary lever for fitting billion-parameter models into limited DRAM and for keeping the weight-streaming working set cache-feasible.

At the extreme end of this spectrum, ternary weights {−1,0,+1}\{-1,0,+1\} achieve 16×16\times compression over FP32, and each weight-activation product reduces to a conditional addition, subtraction, or no-op, eliminating multiplications entirely. This idea has roots in early ternary (li2016ternary, ; zhu2017trained, ) and binary (rastegari2016xnor, ; hubara2016binarized, ) networks, and recent work confirms that ternary LLMs can match FP16 quality: BitNet b1.58 (ma2024era, ) trains models from scratch at ternary precision, while Fairy2i (feng2025fairy2i, ) post-trains existing checkpoints into complex-valued ternary models via quantization-aware training. Fairy2i solves the algorithmic side of the problem; the systems challenge is to translate ternary weights into multiplication-free execution on real hardware.

Existing inference systems, however, do not yet exploit the ternary structure at the microarchitectural level. Fairy2i’s reference runtime, designed primarily as a research prototype for validating model quality, dispatches the quantized weights through standard PyTorch linear operators, leaving room for a dedicated inference engine that eliminates multiplications at the instruction level. Similarly, llama.cpp (llamacpp2024, ) applies dequantization followed by floating-point FMA for its Q2_K and Q4_K_M formats, so the low-bit encoding serves only as a storage optimization while the compute path remains multiply-heavy. An alternative line of work replaces multiplication with bitwise lookup tables: T-MAC (wei2024tmac, ) and BitNet.cpp (bitnetcpp2024, ) use VPSHUFB-based LUT lookups, but the resulting kernels incur indirect memory accesses and non-trivial on-chip LUT footprints that compete with activation data for L1/L2 cache bandwidth, limiting sustained throughput on wide SIMD units.

Our starting observation is that ternary arithmetic on x86 CPUs admits a strictly simpler datapath than either dequantization or table lookup. We pack sixteen {−1,0,+1}\{-1,0,+1\} weights into a single 32-bit word, decode positive and negative masks in a single-cycle _pext_u32 pair (BMI2), and accumulate via AVX-512 masked vaddps/vsubps. The resulting inner loop contains zero floating-point multiplications and zero table lookups, a property we verify through assembly inspection.

Realizing this idea for complex-valued models introduces an additional complication. Fairy2i’s widely-linear representation decomposes each linear layer into eight coupled real-valued sub-GEMVs. A naive schedule launches eight separate parallel regions, repeatedly loads the same activation tiles, spills intermediate buffers to DRAM, and re-decodes related mask patterns, erasing the bandwidth gains from ternary packing. To overcome this overhead, we design a Fused Widely-Linear Kernel that consolidates all eight sub-GEMVs into one row-parallel AVX-512 loop. Four design decisions make fusion profitable: each decoded mask pair serves both real and imaginary accumulation paths, amortizing decode cost (O1); activation registers are loaded once and shared across all eight sub-GEMVs (O2); complex conjugation reduces to a precomputed sign-flipped alias, eliminating per-lane negation (O3); and all partial sums remain register-resident until the final horizontal reduction, removing intermediate DRAM traffic (O4). Roofline analysis confirms that the 16×16\times compression shifts GEMV from deeply memory-bound (AI=0.25{}=0.25) toward the CPU ridge point (AI=8{}=8), yielding a 29.6×29.6\times speedup over FP32, while the same compression provides negligible benefit on bandwidth-rich GPUs.

End-to-end, FairyFuse sustains 32.4 tok/s on a single Intel Xeon 8558P socket, 1.24×1.24\times faster than llama.cpp Q4_K_M, while preserving near-lossless quality: WikiText-2 perplexity is 5.52 (vs. 5.47 FP16) and average downstream accuracy is 66.0%. Assembly analysis confirms zero multiplication instructions in the inner loop, and all measurements are reproducible with CV<{}<{}2%.

In summary, this paper makes three contributions:

  • •

    We implement the first ternary-weight GEMV kernel on x86 CPUs that uses only masked AVX-512 additions and subtractions, with no floating-point multiplications or table lookups in the inner loop.

  • •

    We consolidate the eight sub-GEMVs of a complex-valued widely-linear layer into a single SIMD-friendly loop via mask reuse, input sharing, sign-swap aliasing, and register-resident accumulation, achieving 1.55×1.55\times speedup over unfused execution.

  • •

    Through full-stack evaluation on a 7B-parameter model, we demonstrate that ternary packing makes CPUs, not GPUs, the natural target for extreme quantization, and deliver throughput competitive with 4-bit baselines at 2-bit storage.

Section 3 presents the FairyFuse algorithm and fused kernel design. Section 4 describes the implementation. Section 5 evaluates kernel performance, end-to-end throughput and quality, Roofline analysis, and design ablations. Section 6 discusses related work, and Section 7 concludes.

2. Background and Motivation

2.1. LLM Inference on CPUs

Autoregressive large language model inference (vaswani2017attention, ) is dominated by the decode phase, where each new token is produced by applying a stack of linear layers to a short activation vector. In this regime, the critical operation is a batched matrix-vector product (GEMV) whose cost is set almost entirely by how quickly model weights can be streamed from DRAM. Concretely, generating one token requires touching every weight parameter at least once, so throughput scales with memory bandwidth rather than peak floating-point throughput (pope2023efficiently, ). On commodity CPUs with approximately 200 GB/s of DRAM bandwidth, this bottleneck is especially acute: even aggressively optimized FP16 or INT8 kernels remain far below arithmetic ceilings because the memory system cannot feed operands fast enough.

CPU-based inference nevertheless remains attractive for many deployment settings. General-purpose processors are ubiquitous on laptops, desktops, and embedded systems; they avoid the power, thermal, and binary-stack constraints of discrete accelerators and simplify compliance with data sovereignty requirements by keeping models entirely on the host. Low-latency, privacy-preserving, and fully offline assistants are therefore natural targets for CPU execution, provided that inference throughput can be pushed high enough to be interactive at billion-parameter scales.

2.2. Ternary Quantization for LLMs

Post-training and quantization-aware methods have substantially reduced the precision of weights and activations while preserving model quality. Representative families include post-training quantization for generative transformers (frantar2023gptq, ; yao2022zeroquant, ), activation-aware weight quantization (lin2024awq, ), and accurate 8-bit schemes with careful scaling (xiao2023smoothquant, ). These approaches typically operate at four to eight bits per weight and still rely on multiply-accumulate hardware at inference time.

At the extreme, ternary weights in {−1,0,+1}\{-1,0,+1\} occupy only 2 bits, a 16×16\times footprint reduction versus FP32, and replace each weight-activation product with a conditional add, subtract, or no-op. BitNet b1.58 (ma2024era, ) showed that such ternary LLMs can match FP16 quality when trained appropriately. Closest to our setting, Fairy2i (feng2025fairy2i, ; feng2024ifairy, ) encodes pretrained FP16 models into complex-valued ternary tensors through a lightweight quantization-aware training procedure called PhaseQuant, often requiring only a couple of calibration epochs. Weights live in {1,−1,i,−i}\{1,-1,i,-i\} in the complex domain; equivalently, each real and imaginary component is ternary in {−1,0,+1}\{-1,0,+1\}, yielding the same 2-bit storage budget while exploiting phase structure during training. FairyFuse is designed as the inference engine for these Fairy2i models, converting the algorithmic promise of ternary weights into actual multiplication-free execution.

2.4. Deployment Challenges of Ternary LLMs

Despite the arithmetic structure above, practical systems have not yet translated it into multiplication-free execution on real hardware. Fairy2i’s reference implementation focuses on validating model quality and naturally relies on standard PyTorch operators (F.linear), which dispatch the quantized weights through conventional multiply-accumulate routines; the ternary structure remains unexploited at the instruction level. Alternative low-bit CPU engines such as BitNet.cpp and T-MAC replace multiplications with table-driven decoding (wei2024tmac, ), but LUT kernels trade arithmetic for irregular table lookups that increase L1 pressure and can limit sustained bandwidth on wide SIMD units.

Even when ternary structure is respected, the widely-linear form in Eqs. (1)–(3) introduces a second challenge: each complex layer maps to eight real-valued GEMVs. A naive implementation issues eight separate parallel regions (or kernel launches), multiplying synchronization, loop overhead, and register-spill pressure, and may repeatedly load the same activation tiles from memory for different sub-products. These overheads erode the theoretical gains from ternary arithmetic and complicate integration with existing LLM inference frameworks. The design of FairyFuse directly targets both issues: multiplication-free ternary GEMV on real CPUs and fusion across the eight sub-GEMVs, so that complex-valued quantization becomes a deployment asset rather than a constant-factor slowdown.

3. Design

Figure 2 summarizes the FairyFuse system. The offline stage extracts ternary weights from the Fairy2i-W2 checkpoint and packs them into the 2-bit encoding described below, yielding a 3.3 GB footprint for seven billion parameters (16×16\times smaller than FP32). During online decoding, each generated token passes through 32 Transformer layers; every linear projection that implements the widely-linear map of Eqs. (1)–(3) is evaluated by the FairyFuse fused AVX-512 kernel, which performs all eight real-valued GEMVs as a single row-parallel loop (Section 3.2). We describe the ternary GEMV algorithm first, then present the fused kernel design.

3.1. FairyFuse Algorithm

Let w∈{−1,0,+1}w\in\{-1,0,+1\} be a ternary weight and a∈ℝa\in\mathbb{R} an activation. The product decomposes without any multiply as w⋅a=aw\cdot a=a if w=+1w{=}+1, −a-a if w=−1w{=}-1, and 0 if w=0w{=}0. We pack sixteen weights into one uint32 using two bits (b1,b0)(b_{1},b_{0}) per weight: (1,0)(1,0) encodes +1+1 (add), (0,1)(0,1) encodes −1-1 (subtract), and (0,0)(0,0) encodes 0; the pattern (1,1)(1,1) is unused. This representation achieves 16×16\times compression relative to FP32 while keeping decode logic SIMD-friendly.

Mpos=0xAAAAAAAAu,Mneg=0x55555555u.M_{\mathrm{pos}}=\texttt{0xAAAAAAAAu},\qquad M_{\mathrm{neg}}=\texttt{0x55555555u}.

We obtain 16-bit masks over one 512-bit SIMD lane by

Intuitively, kposk_{\mathrm{pos}} collects the “add” bits and knegk_{\mathrm{neg}} the “subtract” bits after compaction into contiguous sub-registers.

Each 16-element chunk therefore executes zero floating-point multiplications: only bitwise extract, mask logic, and masked add/sub operations touch the execution units.

Comparison to T-MAC.

T-MAC (wei2024tmac, ) rewrites low-bit convolutions into table lookups and feeds lookup results through VPSHUFB, retaining an indirect memory access and a non-zero LUT footprint in cache. Our scheme has no LUT: masks drive SIMD units directly, avoiding gather/scatter-style shuffles on activation-dependent tables and eliminating the associated TLB and cache pressure. Where T-MAC’s bit-wise LUT path issues packed byte shuffles, we issue a single-cycle _pext pair followed by AVX-512 masked add/sub on the live activation vector, a structurally different datapath aligned with ternary {−1,0,+1}\{-1,0,+1\} arithmetic rather than small-integer product tables.

3.2. Fused 8-GEMV Widely-Linear Kernel

A naïve scheduler runs the eight real GEMVs implied by Eqs. (2)–(3) as independent kernels. That design incurs eight OpenMP fork/join barriers, eight redundant scans over 𝐱re\mathbf{x}_{\mathrm{re}} and 𝐱im\mathbf{x}_{\mathrm{im}}, and eight sets of intermediate buffers, each pass re-loading the same activations and re-decoding related mask patterns. FairyFuse collapses all eight into one fused row-parallel loop: each thread owns a batch of output indices ii, walks the inner dimension mm once per row, and updates eight AVX-512 accumulators before emitting yre​[i]y_{\mathrm{re}}[i] and yim​[i]y_{\mathrm{im}}[i]. Four design decisions, described below and annotated as O1 through O4 in Algorithm 1, make this fusion profitable.

The first decision, mask reuse (O1), exploits the fact that each of the four packed weight matrices contributes one (k+,k−)(k^{+},k^{-}) mask pair per 16-element chunk. In the fused loop, every pair is applied to both a real and an imaginary activation path (using 𝐱im\mathbf{x}_{\mathrm{im}} or neg_x_im as required by Eqs. (2)–(3)), so each decode from pp amortizes across two SIMD updates instead of one.

The second, input vector reuse (O2), loads 𝐱re\mathbf{x}_{\mathrm{re}} and 𝐱im\mathbf{x}_{\mathrm{im}} into 512-bit registers exactly once per chunk and shares them across all eight sub-GEMVs, halving activation traffic versus eight independent passes.

Third, the sign-swap trick (O3) handles the terms involving 𝐱¯\overline{\mathbf{x}}, which flip the sign of 𝐱im\mathbf{x}_{\mathrm{im}} in half of the products. We precompute neg_x_im=−𝐱im\texttt{neg\_x\_im}=-\mathbf{x}_{\mathrm{im}} once per layer input and switch pointers inside the inner loop, turning conjugation into a zero-cost alias rather than a per-lane negate.

Finally, register-resident accumulators (O4) keep all eight sets of partial sums in AVX-512 registers for the duration of the inner loop; only after the final chunk do we invoke a horizontal reduction (_mm512_reduce_add_ps) and apply the scalar scales. This removes all intermediate spills to DRAM along the reduction dimension.

Together, these four optimizations yield a 1.55×1.55\times speedup over eight separate GEMVs under DRAM-cold, single-thread measurement, and 1.39×1.39\times at 48 threads where eliminating barriers and redundant parallel regions dominates.

4. Implementation

Table 1 lists the hardware intrinsics that constitute the FairyFuse kernel’s inner loop (intel-intrinsics, ; intel2024sdm, ). The entire GEMV path is built on two arithmetic instructions (vaddps, vsubps) gated by 16-bit masks, plus a single-cycle bit-extraction (pext) for weight decoding; no multiplication or fused multiply-add instructions appear in the critical path. The kernel is compiled with g++ -O3 and the relevant AVX-512 plus BMI2 flags, then exposed as a shared library (libternary_gemv.so). The minimum ISA requirement is AVX-512F + BMI2, available on Intel Skylake-X and later server processors.

Row-level parallelism is exploited via OpenMP: each thread processes a contiguous range of output rows, with no inter-thread synchronization during the inner loop. A single #pragma omp parallel for directive over the output dimension suffices for the fused kernel, eliminating the eight separate fork/join barriers that a naïve implementation would incur. The thread count is set to match the physical core count of a single NUMA node (48 on Xeon 8558P) to avoid cross-socket traffic. All experiments use single-socket NUMA binding via numactl with --cpunodebind=0 --membind=0; cross-socket execution degrades throughput by approximately 10% due to remote DRAM latency (Appendix B).

Packed weights are stored contiguously in row-major order with per-row FP32 scales; the 3.3 GB model file is memory-mapped via mmap (Appendix H). The pipeline comprises approximately 1500 lines of C++ (kernel plus forward pass) and 500 lines of Python, connected through ctypes. Each token traverses 32 Transformer layers: RMSNorm (FP32), the fused 8-GEMV (multiplication-free), and SiLU activation (FP32). The fused GEMV accounts for over 90% of per-layer compute, ensuring the vast majority of arithmetic is multiplication-free (Appendix J).

5. Evaluation

We evaluate FairyFuse along three axes: (1) kernel-level GEMV speedup and assembly verification of the multiplication-free claim, (2) end-to-end throughput and model quality against llama.cpp baselines, and (3) a Roofline-guided analysis explaining why CPUs, not GPUs, benefit from ternary packing.

5.1. Setup

All experiments run on the platforms listed in Table 2. The CPU is an Intel Xeon 8558P with 48 cores and approximately 200 GB/s of DRAM bandwidth. The GPU is an NVIDIA H200 NVL with 140 GB HBM3 and 4.8 TB/s peak bandwidth; it serves as a bandwidth-rich reference point for the Roofline analysis. We compare FairyFuse (Fairy2i-W2, ternary, 3.3 GB) against llama.cpp baselines on LLaMA-2-7B (touvron2023llama2, ): FP16 (13.5 GB), Q4_K_M (4.1 GB), and Q2_K (2.8 GB). Microbenchmarks use 10 warmup iterations and at least 1000 timed iterations; end-to-end throughput is measured over at least 128 generated tokens. All stochastic experiments are repeated with seeds {42,123,2026}\{42,123,2026\}; we report median latency for GEMV and mean±\,\pm\,std for end-to-end throughput. Coefficient of variation stays below 2% across all metrics (Appendix E).

5.2. Kernel Performance

Table 3 and Figure 3 present DRAM-cold GEMV results across three matrix sizes drawn from LLaMA-2-7B linear layers. At equal thread count, the ternary kernel achieves 2.02.0–6.6×6.6\times speedup over FP32, purely from eliminating multiplications and reducing data movement via 16×16\times weight compression. When all 48 cores are utilized, the combined speedup reaches 29.629.6–54.4×54.4\times. The 11008×409611008{\times}4096 case achieves super-linear scaling (54.4×54.4\times) because its packed working set (approximately 174 KB) fits in last-level cache (105 MB), substantially reducing DRAM traffic.

We verify the multiplication-free property via objdump inspection of the compiled kernel. Table 4 summarizes the inner-loop instruction mix: each 16-element chunk executes eight pext (mask decode), eight vaddps (masked addition), and eight vsubps (masked subtraction), with zero vmulps or vfmadd instructions. The only multiplications are 𝒪​(n)\mathcal{O}(n) scalar vmulss for per-channel scale application, accounting for less than 0.025% of dynamic arithmetic.

5.3. End-to-End Throughput and Quality

Having established the kernel-level advantage, we now assess end-to-end performance and model quality.

Table 5 and Figure 4 report the main results. FairyFuse achieves 32.43±0.4132.43\pm 0.41 tok/s, 1.24×1.24\times faster than Q4_K_M (26.15 tok/s) and 1.61×1.61\times faster than Q2_K (20.1 tok/s), while using half the bit-width of Q4_K_M. Notably, FairyFuse remains faster than Q2_K despite Q2_K’s smaller footprint (2.8 GB vs. 3.3 GB), because the multiplication-free datapath reduces compute latency in addition to compressing memory traffic.

In terms of model quality, WikiText-2 perplexity is 5.52 versus 5.47 for FP16 (a gap of only 0.05), better than Q4_K_M (5.68) and substantially better than Q2_K (7.82). Downstream accuracy is 66.0% versus 67.3% for FP16 (−1.3-1.3 pp), outperforming Q4_K_M (65.1%) and Q2_K (56.6%). Per-task breakdowns are provided in Appendix G. The quality preservation stems from Fairy2i’s complex-valued QAT (feng2025fairy2i, ); FairyFuse contributes a lossless, multiplication-free inference path that reproduces these numbers with higher throughput.

5.4. Roofline Analysis: Why CPUs, Not GPUs

A surprising result from our evaluation is that ternary packing, which provides a 29.6×29.6\times speedup on CPUs, actually hurts GPU performance by over two orders of magnitude. Table 6 and Figure 5 explain this asymmetry through the Roofline model.

FP32 dense GEMV has arithmetic intensity AI= 0.25{}\,{=}\,0.25 FLOP/byte, placing it deep in the memory-bound regime on both platforms. Ternary packing raises AI to 8.0 OP/byte. On the CPU (ridge point == 13.5), this shift moves the kernel toward the compute ceiling, unlocking a 29.6×29.6\times speedup. On the GPU (ridge point == 27.9), AI= 8.0{}\,{=}\,8.0 still falls well below the ridge, so the kernel remains memory-bound, and the abundant HBM bandwidth (4.8 TB/s) leaves no headroom for compression to help. Worse, GPUs lack efficient equivalents to BMI2’s _pext_u32, forcing the ternary decode into multiple shifts and masks that degrade ALU utilization. We note that the GPU ternary result uses a custom CUDA kernel with the same algorithmic structure as the CPU version (Appendix F); while further GPU optimization is possible, the fundamental absence of single-cycle conditional add/sub and parallel bit extract in CUDA makes the mismatch structural rather than an implementation artifact.

This analysis positions ternary inference as a structurally CPU-favorable workload: the combination of limited DRAM bandwidth and efficient bitwise ISA extensions (pext, AVX-512 masked ops) makes CPUs uniquely suited for this arithmetic regime.

5.5. Design Ablation

To isolate the contribution of kernel fusion, we compare two configurations: (1) an Unfused baseline that calls the optimized single-GEMV kernel eight times sequentially with separate OpenMP regions, and (2) the FairyFuse Fused kernel that consolidates all eight into one loop.

Figure 6(a) shows that fusion yields 1.55×1.55\times speedup at 1 thread (DRAM-cold) and 1.39×1.39\times at 48 threads, consistent across matrix sizes (Appendix C). The benefit at 1 thread is dominated by O4 (register-resident accumulators eliminate intermediate spills); at 48 threads, O2 (input vector reuse) and barrier elimination dominate.

Figure 6(b) decomposes the fusion gain into individual contributions: O1 (mask reuse) contributes approximately 15%, O2 (input reuse) approximately 20%, O3 (sign-swap) approximately 5%, O4 (register accumulators) approximately 10%, and OpenMP barrier elimination approximately 5%. Each optimization targets a distinct overhead of the widely-linear structure, and their composition is approximately additive.

5.6. Scalability

Thread scaling exhibits three distinct regimes. From 1 to 16 threads, scaling is near-linear with parallel efficiency above 80%, because the ternary kernel’s small working set (packed weights plus three activation vectors) fits comfortably in L3 cache, and each thread operates on independent output rows without synchronization. Between 16 and 48 threads, scaling becomes sub-linear as DRAM bandwidth saturates; adding more cores no longer reduces memory-access latency proportionally. Beyond 48 threads (i.e., spanning both sockets), throughput degrades by approximately 10% due to cross-socket NUMA penalties, where remote memory accesses double the effective latency. The optimal configuration is therefore single-socket binding (48 threads, one NUMA node), which achieves the full 32.4 tok/s. Full thread-scaling curves and NUMA analysis are in Appendix B.

6. Related Works

LLM Quantization Algorithms.

Post-training quantization has become the standard approach for compressing large language models. GPTQ (frantar2023gptq, ) and AWQ (lin2024awq, ) pioneered weight-only compression at four to eight bits with minimal accuracy loss, while SmoothQuant (xiao2023smoothquant, ) addressed activation outliers to enable combined weight-activation quantization. OmniQuant (shao2024omniquant, ) and ZeroQuant (yao2022zeroquant, ) further refined calibration strategies for generative transformers. Mixed-precision regimes such as LLM.int8() (dettmers2022gpt3int8, ) and QLoRA (dettmers2023qlora, ) demonstrated that quality can be preserved at moderate compression by keeping outlier channels in higher precision. More aggressive methods push below four bits: SqueezeLLM (kim2023squeezellm, ) introduced dense-and-sparse decomposition, QuIP (chee2024quip, ) and QuIP# (tseng2024quipsharp, ) exploited incoherence processing and lattice codebooks, and AQLM (egiazarian2024aqlm, ) achieved competitive two- to three-bit compression via additive quantization. At the ternary and binary extreme, BiLLM (huang2024billm, ), PB-LLM (shang2023pbllm, ), OneBit (xu2024onebit, ), and BitDistiller (du2024bitdistiller, ) target one to two bits, while BitNet b1.58 (ma2024era, ) demonstrated competitive quality with ternary weights trained from scratch. On the training side, LLM-QAT (liu2023llmqat, ) and EfficientQAT (chen2024efficientqat, ) proposed data-free and block-wise quantization-aware training for practical low-bit pipelines. All of these methods focus on the algorithmic dimension of quantization; our work is complementary in that, given a ternary-quantized model (Fairy2i (feng2025fairy2i, )), we provide the system that makes multiplication-free inference a reality on commodity hardware.

CPU Inference Systems.

The strongest publicly available CPU inference baseline is llama.cpp (llamacpp2024, ), which provides hand-tuned AVX-512 and ARM NEON kernels across a range of quantization formats including Q4_K_M and Q2_K. T-MAC (wei2024tmac, ) takes a different approach by replacing scalar multiplication with VPSHUFB-based table lookups, eliminating multiplications but introducing on-chip LUT memory pressure and indirect accesses; it reports up to 4×4\times throughput gain over llama.cpp on ARM CPUs (Apple M2 Ultra) at one to two bits. On our x86 platform, the LUT approach faces additional headwinds: Intel’s VPSHUFB operates on 128-bit lanes, limiting throughput relative to our 512-bit masked add/sub path. BitNet.cpp (bitnetcpp2024, ) targets ternary models but similarly relies on LUT-driven approaches. QServe (lin2024qserve, ) co-designs W4A8KV4 quantization with GPU serving kernels for cloud deployment. Our approach is fundamentally different: rather than lookup tables or dequantization, FairyFuse performs direct masked addition and subtraction on packed ternary weights, eliminating both multiplication and table-memory overhead while exploiting the widely-linear structure of complex-valued quantization for kernel fusion.

Efficient LLM Serving.

Efficient inference has been studied from multiple complementary angles. FlashAttention (dao2022flashattention, ) optimizes the attention mechanism via IO-aware tiling to reduce memory reads and writes. vLLM (kwon2023vllm, ) introduces PagedAttention for memory-efficient KV-cache management in serving scenarios. DeepSpeed-Inference (aminabadi2022deepspeed, ) provides multi-GPU inference parallelism at scale. These systems target GPU-centric deployments and focus on scheduling, memory, or attention optimizations; FairyFuse addresses a complementary axis by eliminating arithmetic complexity on CPUs via ternary quantization.

Binary and Ternary Network Foundations.

The idea of replacing multiplications with simpler operations dates back to binary (hubara2016binarized, ; rastegari2016xnor, ) and ternary (li2016ternary, ; zhu2017trained, ) neural networks, as well as the Deep Compression pipeline (han2016deep, ), which combined pruning, quantization, and Huffman coding for efficient storage. These early methods targeted convolutional networks at moderate scale; extending the approach to billion-parameter LLMs with complex-valued quantization (feng2025fairy2i, ; feng2024ifairy, ; trabelsi2018deep, ) and efficient SIMD execution on commodity CPUs is the contribution of FairyFuse.

7. Conclusion

We have presented FairyFuse, an inference system that replaces floating-point multiplication with direct masked addition and subtraction for ternary-weight LLMs on commodity CPUs. By fusing the eight sub-GEMVs of a complex-valued widely-linear layer into a single AVX-512 loop, FairyFuse achieves 32.4 tokens per second on a single Xeon 8558P socket, outperforming llama.cpp Q4_K_M by 1.24×1.24\times while preserving near-lossless quality (WikiText-2 perplexity 5.52, downstream accuracy 66.0%). Roofline analysis reveals that the 16×16\times data compression from ternary packing shifts deeply memory-bound GEMV toward the compute ridge on bandwidth-limited CPUs, making CPUs, rather than GPUs, the natural target for extreme quantization. FairyFuse thus demonstrates a practical, multiplication-free path for deploying billion-parameter LLMs on the processors that are most widely available.

Limitations and future work.

Our evaluation covers a single model (LLaMA-2-7B) and a single ISA (x86 AVX-512). Extending FairyFuse to ARM platforms would require adapting the bit-decode path to NEON’s narrower 128-bit SIMD lanes, and scaling to larger models (13B, 70B) may shift the optimal thread-to-row mapping. FairyFuse currently relies on Fairy2i’s complex-valued quantization; supporting alternative ternary recipes would broaden the range of compatible models. Finally, extending the fused design to batched GEMM for prompt-processing and speculative decoding is a natural next step.

Appendix H Weight Packing Format

H.1. Binary File Layout

Each linear layer is stored as:

[Header: 16 bytes]
  n (uint32):  output dimension
  m (uint32):  input dimension
  scale_re (float32): real scale
  scale_im (float32): imag scale

[Packed weights: n*ceil(m/16)*4 B each]
  U_re, U_im, W_re, W_im

H.2. Packing Details

For each group of 16 ternary values t0,t1,…,t15∈{−1,0,+1}t_{0},t_{1},\ldots,t_{15}\in\{-1,0,+1\}:

(8) packed=∑k=015[𝟙​[tk=+1]⋅22​k+1+𝟙​[tk=−1]⋅22​k]\texttt{packed}=\sum_{k=0}^{15}\bigl[\mathbb{1}[t_{k}{=}{+1}]\cdot 2^{2k+1}+\mathbb{1}[t_{k}{=}{-1}]\cdot 2^{2k}\bigr]

Bit 2​k+12k{+}1 is set for positive weights and bit 2​k2k for negative weights. The _pext_u32 instruction with mask 0xAAAAAAAA extracts all positive indicators; mask 0x55555555 extracts all negative indicators. The resulting 16-bit masks are directly usable as AVX-512 __mmask16 operands.

H.3. Memory Footprint

The embedding and LM head remain in FP16 as they use standard dense operations. The per-channel scales (four FP32 values per row per linear layer) add a small constant overhead but enable significantly better quantization quality than per-tensor scaling.

Appendix I Extended Roofline Analysis

I.1. Arithmetic Intensity Derivation

For a GEMV 𝐲=𝐀𝐱\mathbf{y}=\mathbf{A}\mathbf{x} with 𝐀∈ℝn×m\mathbf{A}\in\mathbb{R}^{n\times m}:

FP32 dense.

Compute: n​mnm FMA =n​m=nm FLOP. Data: 4​n​m4nm bytes (weights) + 4​m4m (input) + 4​n4n (output) ≈4​n​m\approx 4nm bytes. AI =0.25=0.25 FLOP/byte.

Ternary (single GEMV, packed 2-bit).

Compute: n​mnm add/sub =n​m=nm OP. Data: n​m/4nm/4 bytes (packed weights) + 4​m4m (input) + 4​n4n (output) ≈n​m/4\approx nm/4 bytes. AI =4=4 OP/byte.

Ternary (fused 8-GEMV).

Compute: 8​n​m8nm OP. Data: 4×n​m/44\times nm/4 bytes (four weight matrices) + 3×4​m3\times 4m (three input vectors, loaded once via O2) ≈n​m\approx nm bytes. AI =8=8 OP/byte, a 32×32\times improvement over FP32 dense GEMV.

On the CPU, the fused ternary AI of 8.0 approaches the ridge point of 13.5, transitioning from deeply memory-bound to near-compute-bound. On the GPU, AI=8.0{=}8.0 remains well below the ridge point of 27.9, so the kernel stays memory-bound and abundant HBM bandwidth leaves little headroom for compression to help.

Appendix J End-to-End Inference Pipeline Details

J.1. Per-Layer Timing Breakdown

The fused GEMV accounts for 90.8% of per-layer computation, all of which is multiplication-free. The remaining 9.2% (RMSNorm, attention, SiLU) uses standard FP32 operations but is O​(d)O(d) versus the O​(d2)O(d^{2}) GEMV.

J.3. Implementation Details

The inference engine comprises ∼1500{\sim}1500 lines of C++ (kernel + forward pass) and ∼500{\sim}500 lines of Python (weight loading, tokenization, sampling). The C++ code is compiled as libternary_gemv.so and invoked via Python ctypes with zero-copy NumPy array passing. Packed weights are loaded via mmap for lazy page-fault residency management. Total runtime memory: ∼3.5{\sim}3.5\,GB (2.24 GB packed weights + 0.5 GB embeddings + 0.5 GB KV cache + 0.26 GB activation buffers). OpenMP is configured with 48 threads, OMP_PROC_BIND=close and OMP_PLACES=cores.
