# 🤖 AI精读：Voyager: An Open-Ended Embodied Agent with Large Language Models

> **论文信息**
> - **标题**：Voyager: An Open-Ended Embodied Agent with Large Language Models
> - **作者**：Guanzhi Wang, Yuqi Xie, Yunfan Jiang, Ajay Mandlekar, Chaowei Xiao, Yuke Zhu, Linxi "Jim" Fan, Anima Anandkumar（NVIDIA / Caltech）
> - **链接**：https://arxiv.org/abs/2305.16291
> - **一句话总结**：Voyager 是首个由大语言模型（GPT-4）驱动的、在 Minecraft 中具备终身学习能力的具身智能体，通过"自动课程 + 技能库 + 迭代提示"三大核心机制，实现了开放世界中的自主探索、技能积累与知识迁移。

---

## 1. 论文核心问题与研究动机

### 1.1 要解决的核心问题

这篇论文要解决的核心问题是：**如何构建一个能够在开放世界中持续探索、自主学习新技能、并在无需人类干预的情况下不断成长的具身智能体？**

具体来说，这个研究聚焦于一个更明确的场景——在 Minecraft 这个没有预设终点、没有固定剧情的开放世界游戏中，让 AI 像人类玩家一样，从零开始，逐步掌握从基础（砍木头、做饭）到高级（挖掘钻石、对抗怪物）的全套技能。

### 1.2 这个问题为什么重要？

开放世界的终身学习（Lifelong Learning）是人工智能领域的"圣杯"问题之一。现实世界本质上就是开放的——没有人为 AI 预先定义所有可能的任务和目标。一个真正通用的 AI 必须具备以下能力：

- **自主设定目标**：在没有人指定"下一步做什么"的情况下，能够根据当前状态合理规划任务。
- **持续积累经验**：学到的技能不会被遗忘，而是成为未来学习的基础。
- **适应新环境**：在全新的环境中能够复用已有知识，快速上手。

Minecraft 作为实验平台极具代表性：它是一个程序化生成的 3D 世界，拥有多层次的科技树（木制工具→石制工具→铁制工具→钻石工具），需要玩家组合多种技能才能推进。

### 1.3 现有方法的痛点与瓶颈

论文指出，现有方法主要存在三个方向的局限：

**（1）基于强化学习和模仿学习的传统方法：**
- 依赖低层级的运动控制原语（如前后左右、跳跃等），探索效率极低。
- 难以进行系统性探索——智能体经常困在局部区域。
- 可解释性差——学到的策略是黑箱。
- 泛化能力弱——换个环境可能完全失效。

**（2）基于 LLM 的智能体方法（如 ReAct、Reflexion、AutoGPT）：**
- 这些方法原本是为纯文本 NLP 任务设计的，不具备"终身学习"能力。
- 没有技能积累机制——每次任务都从零开始。
- 没有自动课程设计——面对开放世界时不知道该从哪做起。
- 缺乏有效的自我纠错与验证机制。

**（3）LLM 本身的局限：**
- LLM 一次性生成正确代码的能力有限，容易出错。
- 但 LLM 拥有丰富的世界知识和代码能力，如果能合理利用，潜力巨大。

### 1.4 作者的直接动机

作者的出发点很清晰：**GPT-4 拥有强大的世界知识和编程能力，但直接让它做具身控制"一发入魂"不现实——需要一个系统化的框架来引导、纠错、积累。** 于是，Voyager 的设计思路应运而生：用 LLM 的大脑做规划和代码生成，用代码作为动作空间实现复杂行为，用环境反馈和自我验证来迭代改进，用技能库来存储和复用经验。

## 2. 作者提出该方法的思维路径（重点）

### 2.1 第一个观察：为什么已有 LLM 智能体在 Minecraft 中不 work？

作者首先观察到，现有的 LLM-based agent 方法（如 ReAct、Reflexion、AutoGPT）在 Minecraft 中表现很差。根本原因有三：

- **没有任务规划能力**：开放世界没有"下一步该做什么"的提示，这些方法不知道如何推进。
- **没有知识积累**：每次任务都从头来，之前学到的东西没有被保存和复用。
- **一次性代码生成不可靠**：LLM 直接写出的代码经常有 bug，而且没有有效的自我修正机制。

### 2.2 核心直觉：人类玩家是怎么玩 Minecraft 的？

作者从人类玩家的学习过程中获得了关键灵感。人类玩家在 Minecraft 中通常是这样做的：

1. **从简单到复杂**：先学砍树、做木镐，再做石镐、铁镐，最后做钻石镐。
2. **失败了就修正**：做错了看反馈，调整策略再来。
3. **记住成功的做法**：打败僵尸的方法可以复用到打蜘蛛上。
4. **主动探索新事物**：不满足于原地踏步，主动去寻找新资源、新挑战。

这四个特点直接对应了 Voyager 的三大模块设计。

### 2.3 从直觉到框架：三大模块的逐步演化

**第一步：自动课程（Automatic Curriculum）——解决"做什么"的问题**

作者首先意识到，开放世界最缺的是一个"老师"来告诉智能体下一步学什么。传统做法要么靠人工设计课程（费力且不够灵活），要么随机探索（低效且无方向）。作者的创新在于：**让 GPT-4 自己当老师。**

设计思路的演化：
- 最初想到用固定的任务列表——但太死板，无法适应不同的世界生成。
- 然后想到人工设计课程——需要大量 Minecraft 专业知识，且无法应对千变万化的世界状态。
- 最终决定：给 GPT-4 提供智能体的当前状态（装备、背包、周围环境、已完成/失败的任务），让它基于"探索尽可能多样的事物"这个高层目标，自下而上地推荐下一个合适的任务。

这本质上是一种 **上下文学习形式的新颖性搜索（in-context novelty search）**。

**第二步：技能库（Skill Library）——解决"怎么做"的问题**

有了任务之后，智能体需要知道怎么完成。作者观察到：
- LLM 生成的代码经常有错误，一次成功很难。
- 但如果生成过一次成功的代码，就应该记下来，下次遇到类似任务时直接复用。

设计思路的演化：
- 最初想到把成功代码直接存起来——但如何检索？
- 然后想到用自然语言描述作为索引——通过 embedding 在向量数据库中进行语义检索。
- 最终方案：每次任务成功后，将可执行代码存入向量数据库，以 GPT-3.5 生成的程序描述的嵌入向量作为索引键，检索时用"通用建议 + 环境反馈"的嵌入作为查询，返回最相关的 Top-5 技能。

这种设计让技能具有了 **可组合性**——复杂任务可以通过组合已有技能来完成。

**第三步：迭代提示机制（Iterative Prompting）——解决"做得对不对"的问题**

即使有了好的框架，LLM 生成的代码仍然可能有错。作者需要一个闭环的纠错机制。

设计思路的演化：
- 最初想到只用环境反馈——但有些错误不是运行时能检测到的。
- 加入执行错误捕获——代码解释器的错误信息对调试很有用。
- 加入自我验证——用另一个 GPT-4 实例充当"检查员"，判断任务是否真正完成，并给出改进建议。
- 三者合一：环境反馈（中间状态）+ 执行错误（语法/逻辑错误）+ 自我验证（任务完成度检查），形成完整的迭代改进循环。

### 2.4 为什么选择"代码"作为动作空间？

这是一个关键的设计决策。作者没有选择传统的低层级动作（如"向前移动一步"、"跳跃"），而是让 LLM 生成 JavaScript 代码（调用 Mineflayer API）。原因在于：

- **时间扩展性**：一行代码可以封装长时间的行为（如"砍树直到获得 3 个木头"）。
- **可组合性**：复杂行为可以通过调用已有技能函数来构建。
- **可解释性**：代码天然具有语义，人类可以直接阅读和理解。
- **可复用性**：成功的代码可以被存储、检索、组合。

### 2.5 各模块配合的必要性

作者在论文中通过消融实验清楚地表明，这三个模块缺一不可：

- 没有自动课程 → 探索效率暴跌 93%，因为智能体不知道该做什么。
- 没有技能库 → 后期阶段能力停滞，无法构建复杂行为。
- 没有自我验证 → 探索效率暴跌 73%，因为智能体不知道自己是否成功。

这三个模块形成了一个正向循环：**课程引导探索 → 探索产生技能 → 技能支撑更难的任务 → 更难的任务催生更多技能。**

## 3. 算法整体框架概述

### 3.1 整体流程图景

Voyager 的整体工作流程可以概括为以下循环：

```
自动课程 → 提出新任务
    ↓
技能库检索 → 找到相关已有技能
    ↓
GPT-4 生成代码 → 调用环境 API 执行
    ↓
收集反馈（环境反馈 + 执行错误）
    ↓
自我验证 → 判断任务是否完成
    ↓
├── 未完成 → 将反馈送回 GPT-4，迭代改进代码（最多 4 轮）
└── 已完成 → 存入技能库，回到自动课程获取下一个任务
```

### 3.2 输入与输出

- **输入**：GPT-4 黑盒 API 查询（不涉及模型参数微调）、环境观察（通过 Mineflayer API 获取）
- **输出**：一个不断增长的技能库（向量数据库中存储的可执行代码）、探索过程中发现的物品清单、遍历的地图区域

### 3.3 关键特点

- **纯黑盒交互**：不需要访问 GPT-4 的模型参数，不需要梯度训练或微调。
- **终身学习**：技能库随时间持续增长，知识不遗忘。
- **零样本泛化**：在新世界中可以直接复用已有技能库解决未见任务。
- **可解释**：所有行为都是可读的代码，人类可以审查和理解。

## 4. 核心模块与关键机制详解

### 4.1 自动课程（Automatic Curriculum）

**做什么**：为智能体规划探索路径，决定下一步应该尝试什么任务。

**如何工作**：

输入给 GPT-4 的提示包含四个部分：

1. **行为指导指令**：鼓励多样性探索，约束任务难度（"不要超出当前能力太多"）。
2. **智能体当前状态**：背包内容、装备、附近方块和实体、生物群系、时间、生命值/饥饿值、位置。
3. **任务历史**：已完成和已失败的任务列表，反映能力边界。
4. **额外上下文**：由 GPT-3.5 基于当前状态自问自答生成的补充信息（如"我在沙漠中，应该先收集沙子和仙人掌"）。

**关键设计**：课程是 **自下而上** 的，不是预设的固定列表。GPT-4 根据智能体的实际状态和探索进度动态生成任务。这使得课程能够适应不同的世界生成和探索路径。

**温度参数**：自动课程使用 temperature=0.1（而非 0），以鼓励任务多样性。

### 4.2 技能库（Skill Library）

**做什么**：存储和检索可执行的行为代码，实现知识积累和复用。

**存储流程**：
1. 当一个任务被成功完成时，对应的 JavaScript 代码被保存。
2. 由 GPT-3.5 生成该代码的自然语言描述。
3. 使用 text-embedding-ada-002 获取描述的嵌入向量。
4. 将向量作为键、代码作为值，存入向量数据库。

**检索流程**：
1. 面对新任务时，GPT-3.5 先生成一个通用的解决建议。
2. 将该建议与环境反馈拼接，获取嵌入向量。
3. 在向量数据库中检索 Top-5 最相关的技能。
4. 这些技能作为上下文送入 GPT-4 的提示中，帮助生成新代码。

**设计意图**：技能库让 Voyager 避免了灾难性遗忘——之前学过的东西不会丢失，而是在未来遇到类似任务时被自动检索和复用。同时，复杂技能可以通过组合简单技能来构建，实现能力的指数级增长。

### 4.3 迭代提示机制（Iterative Prompting Mechanism）

**做什么**：通过环境反馈和自我验证，迭代改进 LLM 生成的代码。

**三种反馈类型**：

**(1) 环境反馈（Environment Feedback）：**
- 通过 `bot.chat()` 函数获取中间执行状态。
- 例如："我无法制作铁胸甲，因为我还需要 7 个铁锭。"
- 这种反馈帮助 GPT-4 理解程序执行过程中的实际进度。

**(2) 执行错误（Execution Errors）：**
- 来自代码解释器的错误信息。
- 包括语法错误、无效操作（如"Minecraft 中不存在金合欢斧头"）等。
- 这些信息直接用于 bug 修复。

**(3) 自我验证（Self-Verification）：**
- 使用另一个 GPT-4 实例充当"评审员"。
- 输入：智能体当前状态 + 当前任务。
- 输出：任务是否成功完成的判断 + 如果失败，给出改进建议。
- 这比单纯的自我反思（Reflexion）更全面——不仅检查成功与否，还主动提供建议。

**迭代流程**：
1. GPT-4 生成初始代码。
2. 执行代码，收集环境反馈和执行错误。
3. 将所有反馈送入 GPT-4 的下一轮提示。
4. GPT-4 基于反馈修正代码。
5. 重复步骤 2-4，直到自我验证确认任务完成。
6. 如果 4 轮迭代后仍未成功，放弃当前任务，回到自动课程获取新任务。

### 4.4 代码作为动作空间的 API 设计

Voyager 使用 Mineflayer JavaScript API 作为底层控制接口。这些 API 封装了 Minecraft 中的各种底层操作，如移动、挖掘、合成、战斗等。GPT-4 生成的代码本质上是这些 API 的调用组合。

代码的优势在于：
- **时间扩展**：一个函数可以包含数百个时间步的动作序列。
- **层次化**：高级函数可以调用低级函数（如 `craftDiamondPickaxe()` 调用 `mineDiamond()`）。
- **可调试**：错误信息直接映射到代码行。

## 5. 算法相比已有方法的新意与创新点

### 5.1 核心创新：三大模块的系统化整合

Voyager 最大的创新不在于单个模块有多新，而在于 **将三个互补的机制有机整合为一个完整的终身学习系统**。这是以往任何工作都没有做到的：

| 对比维度 | ReAct | Reflexion | AutoGPT | **Voyager** |
|---------|-------|-----------|---------|-------------|
| 任务规划 | ❌ 无 | ❌ 无 | ✅ 子目标分解 | ✅ 自动课程 |
| 技能积累 | ❌ 无 | ❌ 无 | ❌ 无 | ✅ 向量数据库 |
| 环境反馈 | ✅ 有 | ✅ 有 | ✅ 有 | ✅ 有 |
| 执行错误 | ❌ 无 | ✅ 有 | ✅ 有 | ✅ 有 |
| 自我验证 | ❌ 无 | ❌ 自我反思 | ❌ 无 | ✅ GPT-4 验证 |
| 终身学习 | ❌ | ❌ | ❌ | ✅ |
| 代码动作空间 | ❌ 文本动作 | ❌ 文本动作 | ❌ 文本动作 | ✅ 可执行代码 |

### 5.2 创新点逐一分析

**创新点一：自动课程 = 自下而上的自适应任务规划**

已有方法要么没有课程（ReAct、Reflexion），要么需要人工设计（传统 RL 的 curriculum learning）。Voyager 让 LLM 根据实时状态自动生成任务，这在概念上等价于"上下文学习版本的新颖性搜索（in-context novelty search）"。

**为什么有效**：GPT-4 的世界知识使其能够判断"什么是当前阶段合适的挑战"——既不会太难（导致失败），也不会太简单（没有学习价值）。

**创新点二：技能库 = 可组合的程序化知识表征**

已有 LLM agent 方法不保存学到的经验。Voyager 用向量数据库存储可执行代码，通过语义检索实现知识复用。

**为什么有效**：
- 代码天然具有可组合性——`craftDiamondPickaxe()` 可以调用 `mineDiamond()` 和 `craftStick()`。
- 语义检索比关键词匹配更灵活——即使新任务与历史任务描述不完全相同，也能找到相关技能。
- 实验表明，技能库甚至能帮助其他方法（AutoGPT）提升性能——它是"即插即用"的。

**创新点三：三合一迭代提示 = 闭环自改进**

已有的自我反思方法（如 Reflexion）只考虑执行错误。Voyager 的迭代提示结合了环境反馈、执行错误和自我验证，形成了更完整的反馈闭环。

**为什么有效**：三类反馈各司其职——环境反馈告诉你"进度如何"，执行错误告诉你"哪里出错了"，自我验证告诉你"是否真正成功了"。单独使用任何一种都不够，消融实验证明了这一点。

**创新点四：代码作为动作空间**

这虽然不是 Voyager 首创（Code as Policies 等已有相关工作），但 Voyager 在 Minecraft 中系统性地验证了这一思路在终身学习场景下的巨大优势——代码的可组合性和可复用性是传统动作空间无法比拟的。

### 5.3 哪些是核心创新，哪些是配套优化

- **核心创新**：自动课程 + 技能库 + 迭代提示的三模块整合架构。
- **重要设计选择**：代码作为动作空间。
- **工程优化**：使用 GPT-3.5 做成本较低的 NLP 子任务（生成描述、自问自答），用 GPT-4 做核心代码生成和验证；温度参数的细调等。

## 6. 实验设计逻辑：如何证明方法先进且有创新性（重点）

### 6.1 实验平台与设置

- **平台**：MineDojo（开源 Minecraft AI 框架），使用 Mineflayer JavaScript API 进行控制。
- **LLM**：GPT-4 (gpt-4-0314) 用于代码生成和自我验证，GPT-3.5 (gpt-3.5-turbo-0301) 用于辅助任务。
- **嵌入模型**：text-embedding-ada-002。
- **评估维度**：探索能力、科技树掌握、地图覆盖、零样本泛化。

### 6.2 Baseline 选择的合理性

作者选择了三个代表性 LLM-agent 方法作为 baseline：

| 方法 | 特点 | 为什么选它 |
|------|------|-----------|
| **ReAct** | 思维链推理 + 动作生成 | 最基本的 LLM agent 方案 |
| **Reflexion** | ReAct + 自我反思 | 加入了错误学习的改进方案 |
| **AutoGPT** | 任务分解 + 子目标执行 | 最流行的通用 AI agent 工具 |

**对比说服力**：这三个 baseline 覆盖了从简单到复杂的 LLM agent 设计谱系。作者也坦承，没有直接对比基于像素输入的方法（如 VPT），因为控制接口不同，不是同类比较。

### 6.3 主实验结果分析

**实验一：探索能力（Fig. 1）**
- Voyager 在 160 次提示迭代内发现 63 种独特物品，是 baseline 的 3.3 倍。
- **证明了什么**：自动课程有效引导了系统性探索，而非随机乱逛。
- ReAct 和 Reflexion 几乎没有进展——说明没有好的课程设计，LLM agent 在开放世界中寸步难行。

**实验二：科技树掌握（Table 1）**
- Voyager 解锁木制工具的速度比 AutoGPT 快 15.3 倍，石制工具快 8.5 倍，铁制工具快 6.4 倍。
- Voyager 是唯一能解锁钻石工具层级的方法（3 次试验中成功 1 次）。
- ReAct 和 Reflexion 在所有层级全部失败（0/3）。
- **证明了什么**：技能积累 + 课程引导的组合对复杂层次化任务至关重要。

**实验三：地图覆盖（Fig. 7）**
- Voyager 行驶距离是 baseline 的 2.3 倍，穿越了多种地形。
- **证明了什么**：自动课程鼓励探索新区域，而非困在原地。

**实验四：零样本泛化（Table 2）**
- 在清空背包、重新生成世界后，Voyager 仍能在 4 个未见任务上全部成功（钻石镐、金剑、岩浆桶、指南针）。
- 所有 baseline 均完全失败（0/3）。
- **关键发现**：将 Voyager 的技能库提供给 AutoGPT 后，AutoGPT 也能部分解决任务——说明技能库本身就是一个通用资产。
- **证明了什么**：终身学习积累的知识可以迁移到新环境。

### 6.4 消融实验设计逻辑

作者对 6 个设计选择进行了消融：

| 消融变量 | 发现 | 验证了什么 |
|---------|------|-----------|
| 随机课程 vs 自动课程 | 探索效率下降 93% | 自动课程是探索的关键 |
| 人工课程 vs 自动课程 | 人工课程不如自动课程 | 自适应性很重要 |
| 去掉技能库 | 后期阶段停滞不前 | 技能积累推动持续进步 |
| 去掉自我验证 | 探索效率下降 73% | 自我验证是最重要的反馈 |
| 去掉环境反馈 | 效率有所下降 | 环境反馈辅助代码改进 |
| 去掉执行错误 | 效率有所下降 | 错误信息对调试有用 |
| GPT-3.5 替代 GPT-4 | 效率下降 5.7 倍 | GPT-4 的编码能力是质的飞跃 |

**消融设计的逻辑**：每个模块单独移除，观察性能下降幅度，从而量化每个组件的贡献。这种实验设计清晰地回答了"每个模块到底有多重要"的问题。

### 6.5 附加实验：人类多模态反馈

论文还展示了 Voyager 在接受人类反馈后构建 3D 结构（如地狱传送门、房屋）的能力。虽然当时 GPT-4 API 尚不支持视觉输入，但这个实验展示了 Voyager 框架的扩展潜力。

### 6.6 实验设计的不足与局限

客观地说，实验设计也存在一些不足：

1. **环境单一**：所有实验都在 Minecraft 中进行，缺乏其他具身环境（如机器人操控、家庭场景）的验证。
2. **API 依赖**：Voyager 依赖高级 Mineflayer API，与直接从像素学习的方法不是严格对等的比较。
3. **样本量有限**：每种方法仅运行 3 次试验，统计显著性可能不够充分。
4. **成本未讨论**：GPT-4 API 调用成本很高，论文未详细讨论经济可行性。
5. **钻石工具仅 1/3 成功率**：说明在极高难度任务上，Voyager 仍不够稳定。
6. **缺乏长期运行实验**：160 次提示迭代的上限可能不足以展示真正的终身学习潜力。

## 7. 论文贡献、适用场景与局限性

### 7.1 核心贡献

1. **提出了第一个 LLM 驱动的具身终身学习智能体框架**，在 Minecraft 中实现了自主探索、技能积累和知识迁移。
2. **设计了自动课程机制**，让 LLM 自己充当"老师"，根据智能体状态自适应地规划学习路径。
3. **提出了基于向量数据库的技能库**，用可执行代码表征技能，通过语义检索实现知识复用和组合。
4. **设计了三合一迭代提示机制**，结合环境反馈、执行错误和自我验证实现代码的闭环自改进。
5. **系统性地证明了"代码作为动作空间"在终身学习场景下的优势**：可组合、可解释、可复用。

### 7.2 适用场景

- **开放世界游戏 AI**：Minecraft 及类似沙盒游戏中的自主智能体。
- **机器人任务规划**：如果机器人提供代码 API，Voyager 的框架可以扩展到机器人操控。
- **自动化工作流**：自动课程 + 技能库的思路可以应用于自动化任务编排。
- **教育领域**：可解释的代码行为便于教学和研究。
- **AI Agent 研究**：为构建更通用的 AI Agent 提供了系统化的设计范式。

### 7.3 局限性

1. **对 GPT-4 的强依赖**：GPT-3.5 的性能大幅下降（5.7 倍），说明框架对 LLM 能力要求很高。
2. **缺乏视觉感知**：当前仅使用文本观察，无法处理需要视觉理解的任务。
3. **动作空间受限于 API**：只能执行预定义的高级 API 能表达的行为。
4. **代码生成上限 4 轮**：如果 4 轮迭代仍未成功就放弃，可能遗漏一些"差一点就能成功"的任务。
5. **环境反馈依赖文本描述**：复杂的 3D 场景难以用文本完整表达。
6. **缺乏与其他终身学习方法的对比**：如经验回放（Experience Replay）、渐进式网络（Progressive Networks）等。
7. **可扩展性未验证**：技能库的增长是否会带来检索噪声和效率下降？

### 7.4 后续改进方向

1. **融入视觉感知**：结合多模态 LLM（如 GPT-4V）实现视觉-语言联合推理。
2. **降低 LLM 依赖**：使用更小的、专门微调的模型替代 GPT-4。
3. **技能库的主动维护**：对技能进行去重、合并、优化，避免库膨胀。
4. **多智能体协作**：多个 Voyager 实例共享技能库，加速探索。
5. **向真实世界迁移**：在机器人平台上验证框架的有效性。
6. **自适应迭代轮数**：根据任务复杂度动态调整最大迭代次数，而非固定 4 轮。

## 8. 一段通俗中文总结（适合快速理解）

想象你在玩一个全新的 Minecraft 存档——没有人告诉你该做什么，你只能自己摸索。一个普通玩家会怎么玩？他会先砍树、做木镐、挖石头、做石镐……一步步解锁更高级的工具，同时记住成功的做法，下次遇到类似情况直接复用。**Voyager 就是这样一个"AI 玩家"，它用 GPT-4 作为大脑，实现了和人类玩家类似的自主学习过程。**

Voyager 的核心设计非常精巧，由三个互相配合的模块组成：

- **自动课程**：相当于一个"AI 教练"，它看着你当前的装备和背包，告诉你"下一步该试试做铁镐了"——既不会让你做太难的任务，也不会让你原地踏步。
- **技能库**：相当于一个"笔记本"，你每学会一项新技能（比如怎么做铁镐），就把具体步骤记下来。下次再需要做铁镐，或者需要铁做别的东西时，直接翻笔记就行，不用从头想。
- **迭代提示**：相当于一个"试错循环"——写代码→执行→看哪里出错了→改→再执行→直到成功。而且还有一个"检查员"专门判断任务是不是真的完成了。

**实验结果非常亮眼**：Voyager 发现的独特物品是其他方法的 3.3 倍，行走距离是 2.3 倍，解锁科技树的速度最高快 15.3 倍。而且它学到的技能可以带到一个全新的世界里直接用，其他方法完全做不到。

**Voyager 的意义不仅在于 Minecraft**：它展示了一种用 LLM 构建终身学习智能体的通用范式——用代码表示行为、用向量数据库积累经验、用 LLM 自动规划课程。这个框架可以被推广到机器人控制、自动化工作流等各种场景。当然，它也有局限：强依赖 GPT-4 的能力、缺乏视觉感知、对 Minecraft 特定 API 有依赖。但作为一个开创性工作，Voyager 为"LLM 如何赋能具身终身学习"这个问题提供了一个优雅且有力的答案。
